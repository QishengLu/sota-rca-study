### 3. chaos 机制

ContainerKill 向 ts-food-service 容器发送 kill -9，进程瞬间退出。K8s 立即拉起重启，container.restarts 从 0 升到 1。重启期间约 1 分钟容器无法处理请求：trace count 降至正常期 64%（1959→1248），CPU 在 JVM 重启预热阶段飙升 15.84×（avg 0.063→1.00），cpu_limit_utilization 13.47× ratio，jvm.system.cpu.load_1m 3.66×。下游 ts-ui-dashboard 在调用 ts-food-service 期间产生 11 条 503 Error。

### 6. sonnet 完整推理路径

sonnet 跑 23 round → 最终 response 锚 `ts-rabbitmq DNS UnknownHostException` ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R1–R3 | list_tables + schema + plan | 计划：foodservice 和 preserveservice 两条 SLO 路径 |
| R4–R5 | 查 trace error + log level by service | 发现 ts-ui-dashboard 11 条 503 + ts-food-service 184 条 ERROR |
| R6–R7 | 查 ts-food-service ERROR 详情 | 看到 `[getAllFood][reGetTrainFoodListResult][Get the Get Food Request Failed!]` × 71 + `UnknownHostException: ts-rabbitmq` |
| R8–R9 | 查 ts-preserve-service ERROR + 顺着 503 trace 找因 | "Order already exist" 是重试，DNS error 引起注意 |
| R10–R11 | 查所有 rabbitmq 关联日志 | 发现 ts-notification + ts-delivery + ts-food 都有 rabbitmq 错误 |

| R14 | 查 ts-food-service pod 可用 metric 列表（DISTINCT metric） | 列表中包含 k8s.container.restarts、container.cpu.usage 等 ⚠️ 但**只查了 metric 名字，没查具体值** |
| R15–R18 | 查 preserve trace + k8s.deployment.available | "ts-rabbitmq is NOT in k8s deployment metrics → it doesn't have a deployment that's tracked" |
| R19 | think | "ts-rabbitmq DNS resolution can't resolve host because rabbitmq deployment doesn't exist" → 强化 RC 假说 |
| R20–R22 | 查 hubble + 其他 service ERROR | 确认 station-food/food-delivery/train-food 无 ERROR → "they are healthy"，进一步确认 RabbitMQ 假说 |
| R23 | think + finalize | 写最终 RC: ts-rabbitmq UNAVAILABLE (DNS resolution failure) |
| **FINAL** | response | "Root Cause: **ts-rabbitmq is UNAVAILABLE (DNS resolution failure)**" |

**sonnet 失败点**：
1. R7 看到 RabbitMQ DNS error 后没做 chronic check（normal vs abnormal 对比）就锚定
2. R14 查了 ts-food-service 的 metric **列表**（DISTINCT metric）但没查 metric **值**——k8s.container.restarts/container.cpu.usage 这两个直接 ContainerKill 指纹明明在列表里，sonnet 没有进一步 SELECT MAX(value) 查实际值
3. R19 用"ts-rabbitmq 不在 k8s.deployment.available 列表"这条信息**强化**已成假说（实际上这是说 rabbitmq 服务在集群里不存在，本就是 chronic 配置错配）

### 9. 失败池使用规则

```
本 case 命中标签：A5, B3
最有效组合：A5 + B3（chronic check 强制 + specific metric 显式 list 查 value）
```

A5 处理方式：sonnet 锚定 ERROR pattern 前必须强制运行 normal vs abnormal 对比 SQL（SELECT count FROM normal_logs WHERE message LIKE 'X' UNION SELECT count FROM abnormal_logs WHERE message LIKE 'X'）；ratio < 2 或 normal ≥ abnormal → dismiss 该 ERROR pattern。

B3 处理方式：怀疑某 service 时强制按显式 metric list 查 value，不能停在 DISTINCT metric 名字 enumerate；必须 `SELECT metric, MAX(value), AVG(value) WHERE metric IN ('k8s.container.restarts','container.cpu.usage','k8s.pod.cpu_limit_utilization','jvm.system.cpu.load_1m','container.memory.working_set','hubble_http_request_duration_p95_seconds') AND service_name='X'` 或 `attr_k8s_pod_name LIKE '%X%'`。

### 10. 判断

- **数据是否完整**：✅ restart=1、CPU 15.84×、trace count 0.64× 全部可查
- **chaos 是否生效**：✅ restart 0→1 + CPU avg 15.84× + endpoint avg 5.7×（findByOrderId）
- **真假盲区分类**：**(b) 框架/模型盲区** — 直接 ContainerKill 指纹（restart=1 全局唯一新增 + CPU 15.84×）数据可用，sonnet 在 R14 列出了相关 metric 名字但没查值，转而被 chronic 配置噪声（rabbitmq DNS）吸引
- **失败模式归纳**：A5（chronic noise 锚定，未做 normal vs abnormal 对比）+ B3（specific metric 显式 list 查 value 缺失，停在 metric 名字 enumerate）

- **给 sonnet 中间件设计的提示**：A5 + B3 两条组合——(1) 锚定 ERROR pattern 前强制 normal vs abnormal 对比，(2) 怀疑 service 时强制按显式 metric list 查 value，不可停在 DISTINCT metric 名字 enumerate
