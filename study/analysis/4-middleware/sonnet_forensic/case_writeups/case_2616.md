### 3. chaos 机制

chaos-mesh NetworkCorrupt 通过 tc qdisc + netem corrupt 在 ts-route-plan-service pod 的 veth 上对 to ts-travel-plan-service 方向的包做 97% 字节翻转（每包随机字节有 97% 概率被 flip）。直接后果：
- 该方向 TCP 包几乎全部 checksum 失败 → 应用层收到的 HTTP body 损坏 / TCP 重传暴涨
- 业务调用方向：trainticket 业务流是 travel-plan-service → route-plan-service（计算最便宜/最快路线时调路线规划）；direction=to 是 chaos 配置 source→target，但实际 trainticket 该 edge 数据流是 callee→caller 方向 response 包（route-plan 给 travel-plan 返回路径数据），response body 几乎全 corrupt → travel-plan 应用层 Jackson deserialize 失败 / retry / timeout
- caller (travel-plan) 端 wallclock latency 暴涨 + retry CPU 烧；callee (route-plan) 端处理本身正常但 response 几乎都被 drop
- (travel-plan, route-plan) parent-child edge child span 数量大跌（应用层 retry 失败后没生成 child span）

**sonnet 失败点**：
1. **B9 baseline cross-rank 完全跳过**：travel-plan-service ratio_avg=303x（碾压第二名 20x）+ ratio_count=0.02（全表倒数）— 一眼可见 GT，sonnet 看了端点没看 service-level 排序
2. **B1 caller-callee edge ratio 完全跳过**：(travel-plan, route-plan) edge ratio=0.033（-97%）= NetworkCorrupt 决定性指纹，sonnet 没跑这个 SQL
3. **A3 max-only fallacy（GC 维度版）**：用 GC histogram 局部聚合 "1.635s vs 0.265s = 6x" 锚 seat，没看 GC events 数量（normal 27 vs abnormal 3 = abnormal **events 反而更少**，3 events 不构成稳态 GC pressure）
4. **CPU 维度反向信号未识别**：seat container.cpu.usage avg 0.28x **下降** + jvm.cpu.recent_utilization 0.31x **下降**——直接否决"seat 高 CPU"假说，但 sonnet 只看了 max（0.53→0.82）忽略 avg + normal 维度对照
5. **A9 多维度矛盾仍锚**：seat trace count 0.05（跟系统 0.04-0.07 同质，不是特异 silence）+ container.cpu avg 下降 + GC events 反而减少 — 三维度都跟 "seat 是 origin" 矛盾，但 sonnet 仍锚定
6. **chaos 类型先验缺失**：完全没考虑 NetworkCorrupt / NetworkPartition 这种 edge-on-edge chaos 类型（指纹是边级 child span count drop，不是 service-level 单 service CPU 升）

### 10. 判断

- **数据是否完整**：✅ 数据完整，chaos 信号极强且齐全
- **chaos 是否生效**：✅ 完全生效。corrupt=97% 直接体现在 (travel-plan, route-plan) edge ratio 0.033（-97%）+ travel-plan service-level ratio_avg=303x + route-plan jvm.system.cpu 15.28x
- **真假盲区分类**：(b) 框架/模型盲区。数据完整且信号极强（service-level 双扫第一 + edge ratio 决定性指纹），sonnet 跳过 baseline cross-rank + edge ratio 直接深挖单 service 物理指标
- **失败模式归纳**：B9（baseline cross-rank 漏）+ B1（caller-callee edge ratio 漏）+ A3（GC histogram 局部 max 锚定）+ A9（seat 四维度矛盾仍锚）+ A12（cascade 方向反了——上游 303x vs 下游 2.67x，sonnet 没用强弱方向反证）
- **共性 vs sonnet-specific**：
  - **sonnet-specific failure**（qwen3.5 答对 sonnet 答错）。本 case 暴露 sonnet 一个独立失败模式：**直接深挖单 service 物理指标（GC / CPU spike）+ 跳过 baseline cross-rank + 跳过 edge ratio**——这是"用看似精细的 metric 工具构造完整故事，但跳过 RC 定位的 baseline 第一步"
  - 区别于 case 323/3868：那两个是"跑了 baseline 但被 host-level 误读 + 反证不回退"，本 case 是"baseline 根本没跑就直接深挖"
- **给 sonnet 中间件设计的提示**：B9 + B1 + A9 三条组合（详见 Section 9）。核心：**baseline cross-rank（service-level ratio）+ caller-callee edge ratio 必须作为锚定前的硬关卡**——sonnet 倾向于跳过基线扫描直接深挖单 service 工具指标，需要强制每个 case 第一步先跑 SQL 1 + SQL 2
