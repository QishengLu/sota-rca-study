### 3. chaos 机制

ChaosBlade JVMMemoryStress mem_type=1 是 **heap memory 压力**（持续 allocate 大对象不释放，触发 OOM）。注入点 `TravelPlanController.home` 是 controller 入口/健康检查路径（业务热路径调用稀疏，但只要被命中一次就进入 stress 循环）。

预期因果链：
1. heap 持续 alloc → GC 频繁 + JVM CPU 升高 + container.cpu 升高
2. heap 撑满 → OOMError → JVM 退出 → k8s container restart 0→1
3. preserve method-level latency 显著上升（max 触顶 3-4s 的 OOM 窗口请求被卡）
4. caller `ts-ui-dashboard` POST `/travelPlan/*` 端点 503/timeout
5. ts-travel-plan-service 调用的 callee（ts-seat-service / ts-route-plan-service / ts-train-service）被动 cascade（cpu/load 弱升）

跟 case 1280 (mem_type=2 direct memory) 关键区别：mem_type=1 heap 会触发 OOM + restart，是最强 GT 指纹之一。

### 9. 中间件提示维度（针对本 case 4 条失败模式 1:1 设计）

| 失败模式 | 中间件提示维度 | 触发时机 |
|---|---|---|
| A2 | **强证据 sticky + dismiss 必须举证**：当 reflection 中已识别某 candidate 为 RC（明确写出"X is the root cause"或类似措辞）后，任何 dismiss 该 candidate 的论证必须按"chaos 因果链方向"举证——不接受单句反向 rationalize（"could be consequence of retries / could be cascade artifact"）。强证据的判定阈值：specific metric ratio ≥ 5x 同时存在 ≥ 3 个独立维度（cpu / memory / gc / restart / page_faults）→ sticky；dismiss 必须给出 chaos 物理机制反驳（"X 在 chaos 因果链上是 cascade callee，被动接收只会产生 trace duration 升 + IO wait，不可能产生自身 GC 8x，因此被动 cascade 假说不成立 → X 不能 dismiss"）。本 case R18 dismiss 一句话过关 → 必须强制反驳 | reflection 中出现 dismiss 已锚定 candidate 时 |
| A6 + A12 | **host-level vs pod-level 区分 + 节点拓扑反证**：识别 host-level metric（同 node 所有 pods 数值相同：jvm.system.cpu.load_1m / k8s.node.* / 部分 worker-node 共享指标）后，**禁止**把 host-level 信号当 RC 升级——必须再查 pod 自身 specific metric 排除 host 因素后的强度（container.cpu / cpu_limit_utilization 绝对值 + restart）；任一 < 30% / < 2x → host 信号是 host 上其他 pod 的影响，不是该 pod 自身故障。同时强制扫所有 worker node 的 host-level metric ratio——若多个 worker 同样升高（本 case worker2 10.6x + worker5 7.2x）则该信号不能定位单一 node；若 GT 候选所在 node 几乎不变（worker1 1.24x）则 node 假说与候选矛盾 | 候选锚定包含 host-level metric 信号时 |
| A3 + A11 | **trace error 唯一性 + 错率分母强制 + chaos 因果方向校核**：任何 trace 中"唯一/极少 Error span"作 RC 候选时强制：(1) 计算错率分母（Error count / 该 service 总 spans），< 1% 不能当 RC，是 cascade artifact；(2) 沿 chaos 物理机制反向问"chaos 注入点直接受影响的是哪个 service 的什么资源，cascade 链上谁是 caller / 谁是 callee，candidate 在因果链上是上游/下游/旁支"——chaos heap stress on caller 的 cascade callee 被动等待只产生 trace duration 升 + IO wait，不可能产生自身 GC 8x / cpu 5x 强升；(3) candidate 自身 specific metric 排序（container.cpu / cpu_limit_util / restart）必须有 ≥ 1 个维度 ≥ 3x，否则不能锚定 | 锚定基于 "trace 出现 Error/特殊信号的下游" 时 |
| A10 | **state vocabulary 强制小写词表**：state 输出必须从 causal_graph schema 词表选取（小写 `high_cpu` / `high_memory` / `high_gc_pressure` / `high_avg_latency` / `high_error_rate` / `injection_affected` / `restarting` / `timeout` / `healthy` / `unknown`），拒绝大写自创 `HIGH_CPU` / `HIGH_LATENCY` / `HIGH_ERROR_RATE` / `UNAVAILABLE` 等 | final answer 输出前 |

**最有效组合**：A2 + A6+A12——A2 强证据 sticky 单条就能扭转本 case（R18 dismiss 时强制要求 chaos 物理机制反驳，sonnet 给不出 cascade callee 自身 GC 8x 的物理解释）；A6+A12 host-level 反向利用否决断 sonnet 后段 60 round 的 worker2 node-overload 假说追加。两条任一触发都能扭转本 case。

### 10. 判断

- **数据是否完整**：✅ 数据齐全，GT 信号极强（restart 0→1 incident-onset + cpu 8.6x + GC 8x + page_faults 3.76x）
- **chaos 是否生效**：✅ 完全生效（heap stress 标志指纹完整命中：cpu spike + GC 8x + OOM restart + page_faults 飙升）
- **真假盲区分类**：(b) 框架/模型盲区——数据完整且强信号 sonnet 实际看到了，因主动 dismiss 错过
- **失败模式归纳**：A2（强证据 dismiss + 反证不回退）+ A6+A12（host-level 反向利用 + 节点拓扑联想 + 饱和反证缺失）+ A3+A11（trace error 唯一性 + 错率分母缺失 + chaos 因果方向校核缺失，跟 qwen3.5 共性）+ A10（state 自创大写）。详见 Section 8
- **共性 vs sonnet-specific**：
  - **共性**：A3+A11 - 两 model 都从"trace 出现 Error 信号的下游"锚定 ts-seat-service，共享"cascade callee 出现新 Error = RC"的因果归属偏差
  - **sonnet-specific**：A2 - sonnet 在 R17 实际拿到 GT 完整证据 + R18 显式写出 RC 锚定后**主动 dismiss**，qwen3.5 是"没查到"的简单失败，sonnet 是"查到了主动转身"的复杂失败；A6+A12 - sonnet 因为深挖 host-level 信号产生独有的 worker-node 物理拓扑假说，qwen3.5 路径短不到这层
- **给 sonnet 中间件设计的提示**：A2 + A6+A12 组合（详见 Section 9），重点是"强证据 sticky + dismiss 必须举证"——这是 sonnet 在 case 1495 上的根本病灶
