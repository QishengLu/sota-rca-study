### 3. chaos 机制

NetworkBandwidth 在 ts-route-plan-service ↔ ts-travel-service 的网络路径上**双向**注入 token-bucket 限速。两个关键参数决定本 case 信号强度：

- **rate=110 KB/s** 比较严格（case 3236 是 988 KB/s）
- **limit=8519 字节** 较宽松：典型 HTTP request/response body 几百到几千字节，**大部分请求能 burst 通过 limit**，只有偶尔的大 response 被 throttle
- 后果：服务**不会陷入 retry 风暴 + thread 阻塞**（jvm.cpu 不暴涨），只是这条调用边偶尔失败 → trace 没 export，剩下的请求按正常速度走

参考 OBSERVATIONS line 989-1016 已总结的 NetworkBandwidth 信号强度连续谱（按 limit 配置区分）：
- 极端 (< 200 字节)：edge ratio < 0.01, jvm.cpu ~1.0x（case 2715）
- 偏紧 (200-2000)：edge ratio < 0.1, jvm.cpu 1-3x（case 2678）
- 中度 (2000-5000)：edge ratio 0.1-0.3, jvm.cpu 3-15x（case 1140 / 283）
- **宽松 (> 5000)：edge ratio 0.08-0.15, jvm.cpu ~1.0x（不变）— case 4510 在此**

**sonnet 失败点**：

1. **silence-as-RC 反向**（R17 关键转向）：travel2 在 error trace 里没有 span ≠ travel2 unresponsive。chaos 切断上游调用边后下游所有服务都 silent。
2. **没做 caller-callee edge ratio drop SQL**（grep 验证：0 次 `parent_span_id JOIN` SQL）：NetworkBandwidth 唯一可靠指纹（trace 父子调用边 ratio drop）被完全忽略。
3. **没按 attr_source/attr_destination drill hubble metric**（R9-10）：看了 hubble_drop_total 只 service-level，没拆 attr_source / attr_destination 算 edge drop（网络层维度漏跑）。
4. **弱信号 rationalize 假说**（R20+R21 双重加固）：max-only outlier (travel2 两个 1.16s) + 时间 correlation ("CRITICAL correlation") 两个弱信号被用来加固已成型的 silence-as-RC 假说。travel2 distribution avg=40ms / 844 trace 中 2 outlier 不构成显著；时间点重合 vs 340 error trace 也不构成显著。
5. **反证检测后无 fallback 维度**（R8）：正确识别 jvm.system.cpu.load_1m 在多 service 同值是 host-level（这是元认知能力），但识破后**只 dismiss 这个维度**，没换替代维度（应该转 caller-callee edge ratio）。
6. **自创 schema 外 state**（PROCESS_PAUSED + HIGH_LATENCY）：跟 case 3236 重复犯，PROCESS_PAUSED 是 sonnet 跨 case 反复使用的自创标签。
7. **锚 cascade 链外服务**：ts-travel2-service 不在 chaos 边的直接调用链上（chaos 是 route-plan ↔ travel，不是 route-plan → travel2）；qwen3.5 锚 travel-plan 至少在调用链上（cascade 中段）。sonnet 锚得比 qwen3.5 更远。

### 10. 判断

- **数据是否完整**：⚠️ **稀疏**——chaos 宽松配置导致大部分维度在 baseline（jvm.cpu ~1.0x, service avg 0.8-1.0x, log 全 INFO, restart 0），只有 caller-callee edge ratio drop 一个维度有清晰信号
- **chaos 是否生效**：✅ 生效但弱——edge ratio 0.086 vs 背景 0.13-0.14（drop 约 35-40%），cascade 顶层 ui-dashboard avg 5x（timeout 累积），但 GT 服务自身没炸
- **真假盲区分类**：(b) **框架/模型盲区**——edge ratio drop 这个唯一可靠指纹的 SQL 模板已被 OBSERVATIONS line 1016 总结，但 agent 没引导自己跑这种维度的查询；当前 19 类 intent taxonomy 不包含 `edge_drop_detect`
- **失败模式归纳**：silence-as-RC 反向 + caller-callee edge ratio drop SQL 漏跑 + hubble attr drill 缺失 + 自创 schema 外 state（共性 4 条）；sonnet 额外：弱信号 rationalize 假说（max-only outlier + 时间 correlation 双重加固）+ 反证检测后无 fallback 维度 + 锚 cascade 链外服务
- **共性 vs sonnet-specific**：**深度共病**——共性 4 条；sonnet 在共病基础上额外用弱信号 rationalize 加固假说，反证检测正确但识破后无后续动作，并锚到比 qwen3.5 更远的 cascade 链外孤立服务。两 model 都自创 schema 外 state（sonnet PROCESS_PAUSED 跟 case 3236 重复使用，qwen3.5 UNAVAILABLE）
- **给 advisor 的提示**（最高 ROI 三件套）：
  1. **caller-callee edge ratio drop 维度作为 NetworkChaos 类必查动作**（OBSERVATIONS 已列为 M24，本 case 唯一可靠指纹）
  2. **silence-as-RC 反向校准**：候选服务在 error trace silence 时先看是不是上游 caller cut（caller 同样 silent → 真 RC 在 caller 或 caller→callee 边）
  3. **state vocabulary 约束**：掐死 PROCESS_PAUSED / UNAVAILABLE / HIGH_CPU 等自创大写标签，强制回退到 causal_graph 词表
