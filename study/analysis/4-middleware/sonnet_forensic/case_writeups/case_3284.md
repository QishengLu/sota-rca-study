### 3. chaos 机制

chaos-mesh NetworkDelay 通过 tc qdisc + netem delay 在 ts-travel-plan-service pod 的 veth 上对 to/from ts-seat-service 双向流量注入 1609±733ms 排队延迟（packet 在 egress queue 滞留）。物理后果：
- **caller (travel-plan) 端**：HTTP request 到 seat 的 TCP 包出口被延迟 ≈1.6s + response 包 ingress 也延迟 ≈1.6s → caller wallclock 量到的"调用 seat-service"耗时 ≈3.2s。caller 应用层 thread 阻塞等响应，CPU 不烧但 thread pool 占满
- **callee (seat) 端**：seat-service 收到请求 → 处理（~10ms 真实处理时间） → 返回 response。**seat 自己的 server-side span 时长本就不会变**（处理层没被注入），seat 也无任何 log 异常 / metric 飙升
- **决定性指纹**：caller client-span 时长 vs callee server-span 时长**差值** ≈ 2 × delay_duration，是 caller-side / network-layer 注入类 chaos 的双面签名
- 不会让 (caller, callee) child span count 大跌（不像 NetworkCorrupt / NetworkBandwidth），只让 caller wallclock 升

**sonnet 失败点**：
1. **A13 chaos 预期不变信号误当 callee 健康反证（核心失败）**：sonnet R21-R30 自己 reflection 写出"travel-plan POST 3.21s 调 seat-service，seat 响应 10-15ms" — 这恰恰是 NetworkDelay caller-side / network-layer 注入的决定性指纹（callee server 时长本就不会变），sonnet 反过来当作"seat is healthy"排除 seat 候选
2. **B7 caller-callee 双面 span 时差 SQL 没跑**：实测 caller travel-plan Client p50 1391ms vs callee seat Server p50 12.1ms = 不对称比 115x = network delay 双面签名。sonnet 没显式跑这个 SQL（按 span_kind='Client' / 'Server' 拆分），只在单 trace deep dive 偶然观察到
3. **chaos 类型先验缺失**：NetworkDelay 的指纹本就是"caller wallclock 高 + callee server 不变 + edge count 不特异下降"，但 sonnet 没识别此类型先验，把 callee 不动当 callee 健康
4. **A3 max-only fallacy**：用 ConfigRepository.findByName max=3261ms / db.client.connections.use_time max=3244ms 单点 outlier 锚 RC，没看 abnormal avg 仅 8.23ms（99% < 30ms）
5. **B9 baseline cross-rank 不彻底**：travel-plan-service ratio_avg 19.61x（碾压第二名 3 倍）— sonnet 早期看到端点暴涨但没回到 service-level cross-rank 作为 RC 锚定的 sticky 强证据

- **数据是否完整**：✅ 数据完整，chaos 信号极强且齐全（caller-callee 双面签名 + service-level baseline 第一）
