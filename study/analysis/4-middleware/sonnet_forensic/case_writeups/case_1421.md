### 3. chaos 机制

DNSRandom 在 `ts-station-service` pod 内拦截对 `mysql` 域名的 DNS 查询，返回随机 IP。结果：

- station-service 的 JDBC driver 试图连接错误 IP → connect timeout → 重试 → 部分查询命中 / 部分失败
- 失败连接抛 `UnknownHostException` 但**被 JDBC driver 层 catch 后包装成 SQL retry**，应用层 log 完全 silent
- 失败的 SQL 查询根本没创建 OpenTelemetry span → trace 里只剩 "DNS 解析恰好命中" 的成功查询
- 两边都是 GT (`ts-station-service` + `mysql`)，因为 chaos 在 station 网卡，但目标域名是 mysql

四种盲区的难点叠加：
1. **应用 log 0 ERROR**（DNS 异常被 driver 层吃掉）
2. **trace 是幸存者偏差版本**（失败请求 invisible，存活的看上去几乎正常）
3. **service-level trace count -40% 跟全系统 -40% 一致**（不是特异性信号）
4. **唯一直接信号在 cilium hubble L7 metric**：station-service 的 `hubble_http_request_duration_p99_seconds` 8.9 ms → 285 ms (32x)

sonnet 跑 31 round → `ts-config-service` ❌

| Round | 行为 | 关键决策 |

| R24-27 | DB connection pool + GC histogram 加固 | "config GC 4.9x, DB wait 28.5x" → 加固 cause 叙事 |

| R30-31 | timeline + final | "all start within ms" → "common dependency = ts-config-service" |
| 🔴 FINAL | `ts-config-service` HIGH_CPU + HIGH_GC + HIGH_LATENCY ❌ | |

**sonnet 失败点**:

1. **没查 ts-station-service 自己的任何 metric**（grep 全 trajectory 没出现 service_name='ts-station-service' 的 metric query）
2. **hubble 查询用错维度**：用了 `attr_source_workload + attr_destination_workload`（cilium 源/目的 workload 视图），但**没用 service_name 维度** → 错过 station-service hubble_p99 8.9→285 ms 的 32x 信号
3. **R23 自问 "is this cause or effect" 但没验证**：应该 query "abnormal 期 config 上游 caller 数量" 或 "config 调用模式有没换" → 会发现 station 间接通过 basic 调 config 增多
4. **cascade victim CPU 当 cause 的标准陷阱**：jvm.cpu 在排队等响应时也会上升，跟"做更多工作"是同款表现，sonnet 没区分
5. **5-stage breadth 推理但都在 cascade 链路上**：sonnet 跑了 db pool / GC / node assignment / call chain 5 个维度但每个都在 cascade 末端，**没沿 station-service ← basic ← travel 这条线**反向追

### 10. 判断

- **数据是否完整**：✅（hubble metric 完整，trace/log 完整）
- **chaos 是否生效**：✅ 完全生效（hubble p99 32x，jvm.cpu 4.92x，端点 idlist 2.07x avg）
- **真假盲区分类**：**(b) 框架/模型盲区**——数据有 hubble_p99 32x 直接信号，sonnet 查了 hubble 但用错维度（source/dest workload 而非 service_name），qwen3.5 完全没查 hubble
- **失败模式归纳**：
  1. 没查 GT specific metric（hubble per-service）
  2. cascade victim CPU/GC 当 cause（sonnet）
  3. chronic noise 锚定（qwen3.5）
  4. silence-as-health 极端版（应用 log silent + trace count drop 跟全系统一致）
  5. 没识别 system-wide mass drop 反常
- **共性 vs sonnet-specific**：**通用认知盲点**（DNS chaos 类）。sonnet 跟 qwen3.5 锚定的 victim 不同（config 是 cascade，consign 是 chronic）但都没找到 GT。advisor 应优先攻克"hubble L7 per-service 强制查 + system-wide mass drop 识别"两条
- **给 advisor 的提示**：
  1. 当看到全系统 trace count 均匀下降 + 应用 log silent 时，强制 agent query hubble per-service distribution
  2. 当 agent 锚定服务的 CPU/GC 上升时，强制反验证"是 cause 还是 effect"（query 上游 caller 频率）
  3. 当 SEVERE log 是 SQL NonUniqueResultException 类时，先验证 chronic baseline（normal 期是否也有），不要直接当 RC
