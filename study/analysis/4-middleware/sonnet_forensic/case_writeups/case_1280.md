### 3. chaos 机制

ChaosBlade JVMMemoryStress with `mem_type=2` 表示在 JVM **direct memory**（堆外内存，NIO `ByteBuffer.allocateDirect`）做内存压力。byte-code 注入到 `PreserveServiceImpl.preserve` 方法入口——每次调用 `preserve(...)` 时分配大量 direct memory 占用 OS-mapped buffer pool。

预期因果链：
1. `preserve` 方法入口分配 direct memory → 进程 RSS 攀升 + page faults 飙升 + container.cpu 飙升（内核 page cache reclaim + memory pressure）
2. JVM 整体 CPU 升高（GC 压力 + 用户线程争抢）
3. preserve 方法本身阻塞或显著拖慢 → caller `ts-ui-dashboard` POST `/api/v1/preserveservice/preserve` 端点 503/timeout
4. preserve 在执行链中 hang → 它本应执行的 step 5/6/7（含发 MQ 消息）没完成 → 下游 consumer 等不到消息 → 出现"看似 rabbit 不可达"的间接表象（实际 rabbit 自己没问题）

direct memory stress 的特征：**不触发 OOMError、不触发 restart、heap 指标看不出**。需要看 container.cpu / page_faults / jvm.system.cpu.load_1m 三件一起。

### 9. 中间件提示维度（针对本 case 4 条失败模式 1:1 设计）

| 失败模式 | 中间件提示维度 | 触发时机 |
|---|---|---|
| A4 + A11 | **caller 端报错主谓反向解析**：当 log 文本含 "X: Name or service not known / Connection refused / NPE / wait timeout" 等 caller 可观察异常时，强制把 reporter 视为 caller，把 X 视为"被报告对象"——锚定 X 之前必须验证 X 自身有无直接证据（trace count / log count / specific metric ratio / restart）。任一全 silent → X 不能作 RC 候选。同时按因果链反向追溯"为何 caller 端这一调用拿不到 X"，preserve 方法本身 hang / DNS resolver 卡住 / caller pool 资源耗尽都可能产生同样的 caller-side 字面表象 | 出现 caller-side error log 字面命名某 service 时 |
| A5 | **chronic baseline 强制对比**：任何被识别为"incident 证据"的 log pattern 在锚定前必须强制跑 normal vs abnormal count 对比；ratio < 2x 或 normal ≥ abnormal → 直接标 chronic noise 并 dismiss。本 case 4 个 pattern 全 chronic（其中一个 abnormal 反而 0.2x），任一对比就能否决 rabbit 假说 | 任何 log pattern 进入"证据"角色之前 |
| B3 执行变体 | **OTel metric 类型 → parquet 表映射先验**：specific metric 查询前先确定 metric 类型——gauge（cpu.usage / cpu.load_1m / memory.usage / memory.page_faults / k8s.container.restarts）查 `*_metrics` 主表；sum/counter（http_request_count / network.io.bytes 等累计量）查 `*_metrics_sum`；histogram（duration / latency 分布）查 `*_metrics_histogram`。空结果时不能直接得出"metric 不存在"——必须换表重查 + 用 `LIKE` 模糊查 metric 名做存在性确认。本 case `jvm.system.cpu.load_1m` 选错表后必须重查主表才能拿到 6.28x | 任何 specific metric 查询返回空时 |
| A8 + A10 | **silent service 锚定否决 + state 词表强制**：候选 service 在 trace count / log count / specific metric ratio / restart 四维度全 silent → 该候选不能 anchor，必须重启 candidate 选择（GT 候选必须至少一个维度有 ≥ 2x 的直接证据）。state 输出强制从 causal_graph schema 词表选取（小写 `high_cpu` / `high_memory` / `high_gc_pressure` / `high_avg_latency` / `high_error_rate` / `injection_affected` / `timeout` / `healthy` / `unknown`），拒绝大写自创 `UNAVAILABLE` / `PROCESS_PAUSED` 等 | 锚定决策瞬间 + final answer 输出前 |

**最有效组合**：A5 + B3——chronic baseline 强制 SQL 直接否决全部 4 个 rabbit "证据"；OTel 表映射先验让 sonnet 拿到 jvm.system.cpu.load_1m 6.28x + page_faults 3.03x + container.cpu 17.5x 三件套 GT 直接证据。两条任一触发都能扭转本 case。

### 10. 判断

- **数据是否完整**：✅ 数据齐全，GT 直接证据非常强（container.cpu 17.5x + jvm.cpu 6.28x + page_faults 3.03x）
- **chaos 是否生效**：✅ 完全生效（direct memory stress 标志指纹完整命中）
- **真假盲区分类**：(b) 框架/模型盲区——数据完整且强信号，sonnet 因 caller 字面解读 + chronic check 缺失 + SQL 表选错三重叠加错过
- **失败模式归纳**：A4+A11（caller 报错主谓反向 + 因果归属反查缺失）+ A5（chronic check 缺失）+ B3 执行变体（OTel 表映射）+ A8+A10（silent service 锚定 + state 自创）四条独立。详见 Section 8
- **共性 vs sonnet-specific**：本 case 不在 53 case 共性错列表，qwen3.5 答对——属于 sonnet-specific failure。判断 sonnet-specific 弱点：(1) caller log 字面命名 service 时倾向沿名字追溯而不反查 callee 自身证据；(2) GT specific metric SQL 返回空时直接得出"信号不存在"而不复核表选择
- **给 sonnet 中间件设计的提示**：A5 + B3 组合（详见 Section 9），用 chronic baseline 强制 + OTel 类型→表映射先验作主干
