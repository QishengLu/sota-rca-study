### 3. chaos 机制

NetworkBandwidth 用 `tc qdisc tbf` 在 source pod 出站接口上限速。`direction=from` 即 ts-preserve-service pod 发往 ts-travel-service 的流量被压到 0.86 Mbps。

- 物理表现：ts-preserve 调 ts-travel 的 POST 请求 body / response body 在网络层排队，看上去 server-side span 还很短（54-58ms 处理完业务），但 ts-preserve **client-side 等收完 response 字节** 要花数秒。
- 因为 ts-preserve POST `/preserve` 业务流是串行的（Security → Travel → Basic → Order），ts-travel 这一段卡住后**ts-preserve 后续步骤根本不会发出**，导致 ts-ui-dashboard 等 20s（其客户端 timeout）后整条 trace 被 cancel。
- 故双服务 GT：ts-preserve（client 视角的"卡在 travel response 上"）+ ts-travel（chaos target，被限速边的另一端）。

### 6. sonnet 完整推理路径

54 effective rounds → final pred = `ts-basic-service` UNAVAILABLE ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R10-R12 | 看 normal 流程 | 识别 normal preserve 调用链 `Security → Contacts → Travel(848ms) → Basic(31ms) → Order` |
| R20-R23 | 看 baef... 这条 timeout trace | 看到 ts-travel POST 用了 54ms 完成；之后 ts-preserve 没再出 child span |
| R30-R34 | hubble drop 分析 | 注意到 ts-seat hubble drop 8.56 vs 6.27 normal，但 chronic check **dismiss**（present in normal too） |
| R40-R44 | 看其他 cpu/memory metric | 全部正常，dismiss |
| 🔴 R51-R53 锚定瞬间 | 对比 normal vs abnormal trace 子节点 | "在 normal preserve 中，ts-travel 之后下一个 child span 是 ts-basic-service 的 POST；在 abnormal 失败的 baef trace 中，ts-travel 之后 6.35s 没有任何 child span。**所以 ts-basic-service 是 UNRESPONSIVE**" |
| R54 | finalize | "ts-basic-service is UNAVAILABLE — no spans generated from ts-basic-service in any failing preserve trace as a child of ts-preserve-service, while it responds normally when called by ts-travel-service" |

**sonnet 失败点**：

1. **trace 中断点 fallacy**：把"timeout trace 中 X 之后没有 child span"解读为"X 之后调用的 service 不可用"。实际：trace export 是 best-effort，客户端 20s timeout 断开后，ts-preserve 还没发出的 ts-basic 调用根本没产生 span。
2. **缺失即异常 fallacy**：仅以"missing span"为唯一证据，**没用 ts-basic-service 自身的 ERROR log / metric / restart / status_code 做 cross-validation**。如果真的"UNAVAILABLE"，至少应该有 INFO log 大跌 / restart 1 / health-check 失败 / 其他 caller 也调不通。实测全部正常。
3. **edge 全断模式 missed**：sonnet 已经有数据看到 ts-preserve 出站 156→1/3 全断，但只盯着"ts-basic 那一条"，没意识到 ts-preserve 出站到 5 个 callee（travel/basic/seat/order/security/contacts）都跌 -85% 以上 = caller 自己 outgoing 受阻 (NetworkBandwidth 指纹)。
4. **NetworkBandwidth 假设缺失**：54 round 推理中没出现 NetworkBandwidth / network throttling / bandwidth limit 等假设；hubble drop 看了一次但被 chronic check dismiss，之后再没回到网络层假设。
5. **client-vs-server span 混淆**：看到 ts-travel server-side span 54ms 就认为 ts-travel 健康，没意识到 NetworkBandwidth 限制的是 **client 收完 response body** 的时间，server-side processing 仍快。

### 10. 判断

- **数据是否完整**：✅ trace/log/metric 完整，足以诊断（sonnet 自己也拿到所有数据）
- **chaos 是否生效**：✅ 完全生效（ts-preserve→ts-travel edge 跌 98.1%、ts-ui-dashboard POST avg 53x）
- **真假盲区分类**：**(b) 框架/模型盲区** — 数据有充足信号（edge ratio 矩阵、双峰 distribution），sonnet 没自主推 chaos 候选去校核
- **失败模式归纳**：A（missing-span ↔ unavailable 因果误等价 + 缺失反向校核）+ B（client/server 因果方向混淆）+ C（caller-outgoing fan-out 一致性盲区）+ D（chaos 物理机制候选未自主推导）
- **共性 vs sonnet-specific**：A/B/C 是 **sonnet-specific failure**（qwen3.5 在该 case 对了）；D 是**两 model 都偏离的环节**（该 case 仅 sonnet 暴露后果）。失败根源是 sonnet 在 long-trajectory + 细粒度 trace 解构下产生**伪信号锚定 + 不回 chaos 物理机制层校核**——sonnet 独有的"deep-trace-dive trap"
- **给 sonnet 中间件设计的提示**（独立设计，不引用 qwen v1-v4）：A + C + D 三条组合（详见 Section 9 最有效组合），用 chaos 物理机制层先验驱动因果链信号查询，避免统计 ratio 阈值驱动
