### 3. chaos 机制

ts-route-plan-service 在内部 GET 调用多个下游 callee（实测 normal 期 GET 调用对象包括 ts-route-service / ts-travel-service / ts-travel2-service）。chaos 仅拦截**到 ts-travel-service `/api/v1/travelservice/routes/*`** 的 response，注入 3427ms 延迟。其他 GET（如调 ts-route-service `/api/v1/routeservice/routes/*`）完全不受影响。这就形成 caller 端 GET 双峰分布：受影响半部 ~3431ms，未受影响半部 ~9ms。

**Service name confusion 陷阱：** route="/api/v1/**travelservice**/routes/*" 路径包含 "routes"，但目标 service 是 `ts-travel-service`（不是 `ts-route-service`）。同时 ts-route-plan-service GET 也调 `ts-route-service /api/v1/**routeservice**/routes/*`——两个 callee 路径都含 "routes"，service 名也都含 "route"，是命名陷阱多发场景。

### 4. 调用树（normal 期实测）

```
loadgenerator → ts-ui-dashboard → ts-travel-plan-service
  → ts-route-plan-service                     ◀── RC (chaos injected, caller-side)

      ├─ GET → ts-travel-service                 ◀── RC (response delayed 3427ms)
      └─ GET → ts-travel2-service                ← 不受影响

```

### 10. 判断

- **数据是否完整**：✅ caller GET 按 callee 拆分 + 双面 span 对比 + pod placement 数据全部完整
- **chaos 是否生效**：✅ delay_duration=3427ms 完美在 ts-travel-service callee GET p50=3441ms 中体现（误差 0.4%）
- **真假盲区分类**：(b) 框架/模型盲区——数据完整，sonnet 在 R17 已经把 ts-travel-service 列为候选，但选择性只调查 ts-route-service 且被 host-level metric 单维锚定锁死
- **失败模式归纳**：A6（host-level 当 pod）+ A9（多维度反证未做、单 metric 锚定）+ B2（GET span 按 callee 拆分未做）+ B7（caller-callee 双面 span 未对比）
- **共性 vs sonnet-specific**：B2 + B7 是两模型共性盲区（qwen3.5 不依赖即答对）；**A6 + A9 是 sonnet-specific 失败链**——qwen3.5 没追 worker host 歧路，直接用 service-level top 锚定
- **给 sonnet 中间件设计的提示**：A6 + A9 联动（host-level metric 类型识别 + 多维度交叉反证 gate）；这是 sonnet 反复出现的"沿 host-level metric outlier 单维锚定"模式（与 case 1484 高度同源——case 1484 sonnet 锚 worker5 load_1m 263，case 2640 锚 worker1 load_1m 144）
