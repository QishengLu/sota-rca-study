### 3. chaos 机制

跟 case 2836 同款（HTTPResponseReplaceBody caller-side body 整体替换为非法 JSON），区别仅在 caller / callee 配对：
- caller (chaos app) = ts-seat-service
- callee = ts-order-**other**-service（命名陷阱：跟 ts-order-service 极易混淆）
- 路由 = `/orderOtherService/orderOther/*`（包括 /tickets, /refresh, /security 等多个端点）

ts-seat-service 在内部 RestTemplate 调 ts-order-other-service 时收到非法 JSON body → 抛 `HttpMessageNotReadableException: JSON parse error`。同时 ts-seat-service 还调多个其他下游（含 ts-order-service 的 /orderservice/order/tickets），这些调用本身不受 chaos 影响，但 ts-seat-service 因 JsonParse 异常风暴让 thread pool 紧张，**少量调 ts-order-service 的请求碰到 thread/socket 资源不足而抛 Connection reset**——这是少数派 cascade 副作用日志。

### 10. 判断

- **数据是否完整**：✅ chaos 直接指纹在 SEVERE log message 中清晰可见（118/119 同质 JsonParseException）
- **chaos 是否生效**：✅ caller 大量同质 JSON parse 错误，server 端 0 errors—— HTTPResponseReplaceBody caller-side 标准签名
- **真假盲区分类**：(b) 框架/模型盲区——数据中信号清晰（B10 dedup 立刻可见），但 sonnet 用 sample LIMIT 命中少数派 + 命名陷阱 + chronic noise 三层叠加误锚
- **失败模式归纳**：B10（SEVERE dedup 未做）+ A4（log 提及 service 字面当 RC + 命名陷阱）+ A5（rabbitmq chronic noise 锚定）+ A9（多维反证已观测但未触发回退）+ A10（自创 state 词）
- **共性 vs sonnet-specific**：本 case 不在 53 case 列表，qwen3.5 forensic 不可达——但**与 case 2836 失败模式 1:1 重复**，可视为 sonnet 在 HTTPResponseReplaceBody 类 chaos 上的稳定 systemic failure（B10+A4+A9+A10 核心链 + A5 本 case 额外）
- **给 sonnet 中间件设计的提示**：与 case 2836 强烈呼应——HTTPResponseReplaceBody 类 chaos 应在中间件维度专门设计 dedup-first 子流程，且配合 A4/A5/A9/A10 多维 gate；本 case 的存在证明 case 2836 的失败模式是 sonnet 在该 chaos 类型上的**稳定模式**，不是单 case 偶然
