### 3. chaos 机制

JVMException 通过 ChaosBlade 字节码增强 `OrderOtherServiceImpl.queryOrdersForRefresh`：每次调用直接抛 `java.lang.RuntimeException("Unexpected error")`。异常发生在 ts-order-other-service 自身代码内部，跟下游或其他 service 无关。

因果链：UI 调 `POST /orderOther/refresh` → controller 调 service.queryOrdersForRefresh → 该方法**内部直接抛 RuntimeException**（chaos 注入）→ Spring DispatcherServlet 捕获并记 SEVERE log → 返回 HTTP 500。

GT 单服务：`ts-order-other-service`。

### 6. sonnet 完整推理路径

sonnet 跑 125 msg（62 round）→ `ts-seat-service` ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R3-10 | 查 abnormal endpoint error rate | **🟢 正确锚定**：ts-order-other-service `POST /refresh` 497/498 Error |
| R10 reflection | think | "POST /api/v1/orderOtherService/orderOther/refresh has 497/498 spans with errors" |
| R15-22 | 查 ts-order-other-service log | 看到 SEVERE `RuntimeException: Unexpected error`；显式 think "the refresh failure doesn't seem to have downstream calls" → 即认知到这是 internal exception |
| R28-32 | 全 cluster 扫 memory | 发现 ts-seat-service memory 1.9GB（其他服务 0.85GB）→ "huge outlier" |
| 🔴 R32 锚定瞬间 | think | "ts-seat-service has a dramatically higher memory usage (1.9GB avg, 1.97GB max). This is a huge outlier" → 切换调查目标 |
| R35-40 | 验证 ts-seat-service normal vs abnormal memory | normal 0.85GB → abnormal 1.9GB（2.26x） |
| R45 reflection | think | "ts-order-other-service: 497 SEVERE errors - RuntimeException... **The error is not related to downstream services**" → **即使认识到 internal**，仍未撤销 ts-seat-service 假说 |
| R55-58 | 查 ts-seat-service GC + JVM memory | GC duration 6x；JVM heap used max 1.2GB（limit 2.15GB）|
| R60 timeline | 查 memory 时间序列 | 看到 06:18→0.75GB, 06:23→1.92GB 上升 → 强化"memory leak"假说 |
| 🔴 R62 final | 输出 RC | `ts-seat-service` + state=['HIGH_MEMORY','HIGH_GC_PRESSURE']（实测 0% error + 0 SEVERE + 61% memory limit 未 OOM）|

**sonnet 失败点**：
1. R20-R22：正确识别"internal exception in ts-order-other-service"但未 sticky
2. R32：切换到 cluster-wide memory scan → 被 ts-seat-service 单一 metric outlier 吸引
3. R45：再次承认"error not related to downstream services"，仍不回退到 ts-order-other-service
4. R62：输出与 chaos 注入点无因果关系的 ts-seat-service，且自创 state 标签

### 10. 判断

- **数据是否完整**：✅ 497 SEVERE RuntimeException + endpoint Error 是 JVMException 教科书指纹
- **chaos 是否生效**：✅ 497/498 ≈ 99.8% queryOrdersForRefresh 调用受影响
- **真假盲区分类**：**(b) 框架/模型盲区** — 信号充分；SEVERE log 直接命名 service + 异常类型，sonnet 自己也认知到是 internal exception
- **失败模式归纳**：A3（memory ratio outlier 单维度锚定）→ A12（cluster pressure 反向因果合理化）+ A2（497 SEVERE 强证据未 sticky）+ A9（多维度反证仍锚）+ A10（自创 state 标签）
- **共性 vs sonnet-specific**：全部 sonnet-specific
- **给 sonnet 中间件设计的提示**：A2 + A3 组合是本 case 最高 ROI 干预点；A2 需要"≥ 100 SEVERE 异常 log 服务必 sticky 进 RC，后续假说不能完全替换"；A3 需要"memory ratio outlier 锚定前必先验绝对值（memory_limit_utilization < 80% / heap used < heap limit 80%）+ 因果路径（候选 service 是否在 chaos 注入位点的调用拓扑上）"。
