### 3. chaos 机制

NetworkBandwidth chaos 在 `ts-basic-service ↔ ts-preserve-service` 之间施加双向带宽限制（rate≈520kbps），**两端 pod 的网络吞吐被极度压低**。在 trainticket call chain 中：

- `ts-ui-dashboard → ts-preserve-service`（入口）会 timeout（loadgenerator 20s 超时）
- `ts-preserve-service → ts-basic-service`（preserve 阶段调 basic 查 travel 信息）的请求大部分发不出去
- `ts-basic-service → ts-travel-service` 路径（来自 querySideTransfer 等其他入口）也受影响因为 ts-basic 自身网卡受限

GT 同时标 ts-basic-service 和 ts-preserve-service 因为带宽限制是边级（**两端 pod 都受影响，不能单独归因到一边**）。

### 6. sonnet 完整推理路径

sonnet 跑 130 round → `ts-seat-service` ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R1-R5 | think + schema discovery | 摸清表结构 |
| R6-R10 | preserve 端点查询 | 发现 preserve 入口 20s timeout，loadgenerator/ts-ui-dashboard span 异常（这步是对的） |
| R11-R20 | trace 内部穿透 | 沿 trace `dbb0c808...` 看 preserve 内部，发现 preserve→security/order 等下游 |
| R21-R25 | 端点消失对比 | "Normal trace: preserve `POST /preserve` 156 次，每次调 ts-seat-service `POST /seats` create；Abnormal trace: ts-seat-service `POST /seats` 完全 0 次" |
| 🔴 R26-R30 锚定瞬间 | think | "**ts-seat-service POST /seats 在 abnormal 完全消失** — 这是关键发现！seat create 是 preserve 的关键步骤，没了就走不下去" |
| R31-R40 | 验证 ts-seat metric | 查 ts-seat container.cpu.usage 0.07x 偏低，jvm.cpu 0.000135 偏低，logs 只有 left_tickets 没 create — sonnet 把"低使用率 + 没 create log"当 RC 证据 |
| R41-R45 | hubble TCP flag | 发现 ts-preserve→ts-seat TCP flags 6397→XXX 大跌（cascade，但 sonnet 解为 ts-seat 网络问题） |
| R46-R55 | worker3 节点定位 | 把 ts-seat 在 worker3 当线索，做 worker3 系统负载分析 |
| R56-R130 | rationalize | 用 worker3 的 system load 数据反复合理化 ts-seat-service "RC" 假设 |
| R130 FINAL | output | `ts-seat-service` HIGH_LATENCY ❌ |

**sonnet 失败点**：
1. **silence-as-causation 倒因果**：把 `ts-seat-service POST /seats` 156→0 当 RC 证据，没意识到这是 cascade（preserve 卡了，自然不调 seat create）。**入口端点 156→0**(preserve POST/preserve) **vs 被调端点 156→0**(seat POST/seats)在物理上是 caller 被堵 vs callee 被卡的镜像，但 sonnet 没区分。
2. **没看 SQL 1 service-level avg ratio distribution**：如果看了，ts-seat ratio_avg 0.9x（normal）vs ts-preserve trace count 0.05x + ts-ui-dashboard avg 10.9x 的对比立即露出 ts-seat 不是 RC。
3. **没做 caller-callee edge ratio drill**：NetworkBandwidth 极端版的标志性指纹 `ts-preserve→ts-basic 156→0` (ratio 0.000) 完全没查；sonnet 的 R6-R20 都在沿单条 trace 内部 drill，没做横向 edge-level 比较。
4. **chronic check 不对称**：sonnet 验证了 ts-seat-service 自身 metric "low CPU 0.07x"，但**它没把 0.07x 解读为反向证据**（cpu 低意味着自己没在工作 = 自己没事），反而当成"ts-seat 异常停摆"。
5. **trace 中断点 fallacy 变体**：把"preserve→seat 调用消失"等同"seat 故障"，但实际是 caller 被堵导致 callee 没流量。
6. **anchor 锁死后 rationalize 80+ rounds**：R26 锚定后，剩下 100 个 round 几乎全在合理化（worker3 节点、hubble drops、TCP flag 大跌都被解为 ts-seat 故障），没回头质疑 anchor。
