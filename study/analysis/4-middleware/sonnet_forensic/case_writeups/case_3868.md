### 3. chaos 机制

ChaosBlade JVM agent (ByteBuddy) attach 到 ts-config-service JVM，给 `deleteConfig` 方法注入 601ms 延迟。**问题**：trainticket 业务流（loadgen 模拟用户买票 / 改签 / 取消）**不会调用** `deleteConfig`（只读 / 创建 业务，不删配置）。

**实测**：`deleteConfig` 类 span 在 normal=0 + abnormal=0 — chaos target 方法完全不被触发，chaos 物理上无效果。归 `unsolvable:chaos-not-effective`，与 case 339 / 2130 同类。

### 10. 判断

- **数据是否完整**：⚠️（abnormal 数据完整，但 chaos 主信号物理上不存在）
- **chaos 是否生效**：❌ **完全未生效**。`deleteConfig` 在 abnormal 期 0 span。归 `unsolvable:chaos-not-effective`（与 case 339/2130 同类）
- **真假盲区分类**：(c) chaos 未生效。**理论上仍可命中 GT**：识别 chaos-effectiveness gate 后，回查 incident 描述端点的 ownership service（travelPlan 类业务必调 config-service 取配置）+ 加 chaos 副作用弱三件套（jvm.cpu 1.67x + hubble NaN + filesystem 6.48x）作保底
- **失败模式归纳**：A6（host-level 误读）+ A12（worker-node "node-level saturation"假说但 pod 资源未饱和）+ A5（chronic RabbitMQ 锚定）+ A11（"Connection refused"字面当 RabbitMQ down）+ B9（baseline cross-rank 漏扫 GT 弱信号）+ 新模式 chaos-effectiveness gate
- **共性 vs sonnet-specific**：
  - **共性**：都没识别 chaos-not-effective gate（全表 noise floor + chaos target span 0 次）+ 都没扫 GT 弱副作用三件套
  - **sonnet-specific**：sonnet 跟 case 323 完全相同的 A6 + A12 联合失败模式重现——跑了 host-level 拆分但用错方向（worker4 节点说），且这次叠加了 A5（chronic noise 锚定）+ A11（字面解读 RabbitMQ）多层 rationalization。这是**跨 case 跨故障类型 sonnet 重复出现的特征失败模式**：sonnet 倾向于用 worker-node 物理拓扑联想构造看似系统级的解释，掩盖单一 pod 级反证不足
- **给 sonnet 中间件设计的提示**：chaos-effectiveness gate + A6 + A5 三条组合（详见 Section 9）。核心：当全表 ratio noise floor 时，应触发"chaos-not-effective gate + incident-端点 ownership service 保底"两步走；A6 host vs pod 区分在 sonnet 上是高频失败模式，需要硬关卡（同 host 多 pod 同值即判 host-level，禁止用此 metric 单独锚 pod RC）
