### 3. chaos 机制

`direction=from` + `source=ts-food-service`，意味着限制 ts-food-service 出向（response 方向）到 ts-ui-dashboard 的带宽。调用关系是 ts-ui-dashboard → ts-food-service（GET /foodservice/foods/...），response body 几 KB 但 limit 仅 2596 字节，导致 response 分多包慢传 → ts-ui-dashboard 端等待 → 共病：

- ts-food-service 自己 server 端响应正常但 outbound 网卡限速，jvm.cpu.load_1m 飙升（D-state thread 等 socket buffer）
- ts-ui-dashboard 作为 caller 在等 food response，端点 wallclock 延迟暴涨

中度配置（limit 2596）让大部分 small response 仍能通过，所以 service-level 整体只是温和劣化（service avg 1.5-1.7x），但**端点级**和 **specific metric** 都有强信号。

### 6. sonnet 完整推理路径

sonnet 跑 38 round → 锚 `ts-seat-service` HIGH_CPU ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R3 | discovery | 拿 schema |
| R5-9 | service-level latency | 识别 cascade（loadgenerator/ui-dashboard/preserve/travel/basic 等普遍劣化），但**没**单独 drill ts-food-service 端点 |
| R11 | 排 noise | dismiss verification-code WARN（chronic）+ ts-order-service WARN |
| R13 | trace drill | 看 ts-travel-service `POST /trips/left` avg 318ms，但走偏到 travel/seat/security 分支 |
| R15 | metric_scan | 用 `LIKE '%cpu%'` 等宽泛 metric，得 +1015% admin-order 但绝对值小 → dismiss |
| R17-19 | hubble drop | 发现 hubble_drop_total +26%（cluster-level）→ 判 "moderate" 跳过；**没分 service 跑 hubble_http_request_duration_p95** |
| R21 | trace 单点 outlier | 看 6.68s slowest trace，发现 ts-security-service / ts-order-service 单 span 2.9-3.5s |
| R26 | 同 | 锁定"几个 trace 中段 service span 极慢" |
| R28 | "CRITICAL" worker5 联想 | 发现 ts-order/travel/seat 都在 worker5 → 跳到"node-level CPU 资源争抢"假设 |
| 🔴 R30 锚定瞬间 | metric drill 锁 ts-seat-service | "ts-seat-service +60% CPU 增加（0.486 vs 0.304）" |
| R32 | max-only 加固 | "**HUGE DISCOVERY** ts-seat-service CPU peak 1.474 cores!" — 用 max 单点值当主证据 |
| R34 | 自我矛盾未纠正 | "cpu_limit_utilization 仅 9.7%" → 自己发现 CPU 没饱和，但**没放弃假设**回 hubble/网络层 |
| R36 | 加固 | "peak 29.5% not near limit" → 仍坚持 ts-seat-service |
| R38 + FINAL | 锁定 | ts-seat-service HIGH_CPU |

**sonnet 失败点**：
1. **Worker-node noisy-neighbor 联想**：发现 3 个 service 同节点 → 跳到"CPU 争抢"假设，注意力被锁死
2. **Max-only fallacy**：用 ts-seat-service CPU peak max 1.474（单点）当主证据，没看 distribution（avg 仅 0.486）+ cpu_limit_utilization 仅 29.5% 自相矛盾未触发回退
3. **dismiss 网络层信号过早**：R17-19 看到 hubble_drop_total +26% 集群级，判"moderate"跳过；**没分 service 跑 specific metric**（jvm.system.cpu.load_1m / hubble_http_request_duration_p95），如果跑了会看到 ts-food-service 6.21x / 5.53x
4. **没做 caller-callee edge ratio drop SQL**：NetworkBandwidth 类 chaos 的标志性维度全程没查（即便边 ratio 中度也能浮现 GT chain）
5. **没看 (service, span) 端点 distribution**：ts-ui-dashboard 端点 GET /foodservice/foods/.../Z1235 ratio 12.5x 是 GT 直接证据，sonnet 看了 trace outlier 但没系统跑端点分位数 SQL
6. **silence-as-health 反例**：ts-seat-service ERROR/SEVERE log 全 0，但 sonnet 还是锚它当 RC
7. **host-level metric 误读**：ts-seat-service jvm.system.cpu.load_1m 13.64x 跟 ts-consign-service 完全相同（11.135/151.832 一致），证明是同节点 host-level，但 sonnet 当 pod-level 信号

### 7. 跨模型快速结论

| 项 | sonnet | qwen3.5 |
|---|---|---|
| 最终 pred | `ts-seat-service` HIGH_CPU ❌ | `ts-consign-service` HIGH_ERROR_RATE ❌ |

### 10. 判断

- **数据是否完整**：✅（hubble_p99 个别 NaN 但 p95 / load_1m 充分）
- **chaos 是否生效**：✅（jvm.cpu.load_1m 6.21x，hubble p95 5.53x，端点 12.5x，强生效）
- **真假盲区分类**：**(b) 框架/模型盲区**——数据有强 GT 信号（jvm/hubble/endpoint 三个维度都有），sonnet 没查 specific metric，没做 edge ratio，没看端点分位数
- **失败模式归纳**：worker-node noisy-neighbor 联想 + max-only + host-level metric 误读 + silence-as-health 反例 + dismiss 网络层信号过早
- **共性 vs sonnet-specific**：**通用认知盲点**——advisor 优先攻克目标。两 model 都没系统查 specific metric / edge ratio / endpoint distribution。但 sonnet 特有"worker-node 物理联想 + 自相矛盾不回退"（这个 sonnet 比 qwen3.5 更严重）
- **给 advisor 的提示**：
  1. specific metric 必查清单（jvm.system.cpu.load_1m / hubble_p95/p99 / container.cpu.usage / k8s.container.restarts）作为"无条件触发"动作
  2. caller-callee edge ratio drop SQL 升级为 NetworkChaos 类怀疑下的"必查动作"
  3. (service, span) 端点 distribution drill：当 service-level signal 不显著但仍怀疑时强制执行
  4. silence-as-health 反向校准：锚定时强制验证 anchor 自身的 ERROR/status_code/specific metric
