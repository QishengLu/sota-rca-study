### 3. chaos 机制

NetworkBandwidth chaos 在 `ts-security-service` 这一侧的网络出口（client → mysql:3306）注入一个 token-bucket：rate ≈ 988 KB/s + buffer 8226 字节。这意味着：

- security 服务的 JDBC client 可以建立 TCP 连接（控制面 SYN 包很小，挤得过去），但任何 SELECT 请求一旦 payload 超过 buffer 容量，TCP 就只能按 988 KB/s 的速率慢慢漏出去
- 实际 RTT 被拉到 26-30 秒（abnormal SELECT span p50 ≈ 26830ms）
- mysql 端长时间收不到下一个 packet → 触发 mysql 自身的 `wait_timeout` → 输出 `[Note] Aborted connection N to db: 'ts' ... (Got timeout reading communication packets)`，然后从 server 侧主动关闭
- security 的 HikariPool 把这种被动关闭识别为 `SQLSTATE 08S01 Communications link failure`，把连接 mark broken
- 因此 **client-side 报错的根因在 client 自己的出口 buffer**，不是 mysql server

GT 字符串包含 `mysql` 是因为 chaos 的注入边涉及 mysql，但 causal_graph 把唯一 root_cause 标在 `service|ts-security-service`——mysql 本身是健康的。

| R10 | think_tool | "Database connection failures... MySQL communications error" — **正确识别证据**，但下一步走偏 |

| FINAL | 输出 root_causes=[mysql-0 PROCESS_PAUSED] | judge: "Root cause services ['mysql-0'] do not match correct answer(s)" |

**sonnet 失败点**：

1. **container-resource-healthy = service healthy fallacy**（R15 关键 dismiss）：
   只看 container.cpu / memory / network.errors / restart，没查应用层 jvm metric。jvm.system.cpu.load_1m 2.14x（线程堆积）原本是这种"service 卡在 IO 等待"的最强指纹——sonnet 整 26 round 没有任何一次查 jvm.* metric（grep 验证 0 次）。

2. **CPU counter idle = process paused 自创 fallacy**（R23/R25）：
   `container.cpu.time` 是 cumulative monotonic counter；进程闲着（没干活）counter 自然不变。"counter 不增长"等价于"进程没干活" ≠ "进程被外力暂停"。真正的 process paused 信号是 `k8s.pod.phase ≠ Running` 或 restart 增长——这两都健康。sonnet 创造了一个数据驱动的 state，但 causal_graph 节点的 state list 里根本没有 `PROCESS_PAUSED`（schema 中只有 high_avg_latency / high_p99_latency / high_error_rate / timeout / injection_affected / unknown）。

3. **silence-as-victim**（mysql idle = 受害者）：
   mysql CPU ↓77% 不是因为 mysql 被冻结，而是 client（security）的出口 buffer 卡住，请求过不来。idle ⇒ 健康（在等 client）；sonnet 反向解读为 idle ⇒ 被外力暂停。

4. **Communications link failure 因果倒置**：
   client 端报"Communications link failure / packet not received from server N ms ago"，sonnet 把"server 报错说收不到对面" 误读为"server 那边停了"。实际上这是 client 出口被掐 → server 主动 timeout 关闭连接 → client 看到连接断 → client 报错。报错的措辞虽然指向 server，但根因在 client 的出口。

5. **没做 service-level avg duration ratio 比较**：
   ts-security-service 1902x vs ts-preserve-service 286x vs ts-ui-dashboard 8.7x。1902x ≫ 286x = security 是源头。sonnet 没做这个 cross-service 大小比较（只看了单个 service span 平均，没把所有 service 排序）。

### 10. 判断

- **数据是否完整**：✅ 完整（端到端 ratio + jvm.cpu.load + JDBC SEVERE + mysql 健康 metric 全部齐全）
- **chaos 是否生效**：✅ 强生效（端到端 1902x，DB span 20000-50000x，client 端 18 条 JDBCConnectionException）
- **真假盲区分类**：(b) **框架/模型盲区**——数据有强信号（jvm.cpu.load 2.14x、service-level ratio 1902x、SEVERE 18 条都没歧义），但 sonnet 没引导自己查 jvm 应用层 metric，并且自创 state 标签
- **失败模式归纳**：container-resource-healthy 误 dismiss + counter idle = paused 自创 fallacy + client-side connection-error 因果倒置 + 自创 PROCESS_PAUSED state
- **共性 vs sonnet-specific**：**sonnet-specific failure**——qwen3.5 同 case 答对，其差异不在数据观察能力（两者都漏查 jvm 和 ratio），而在 sonnet 的"深推理 + 自创因果"倾向把它从一个本来已经识别正确的客户端报错（R10）推到了一个 schema 外的服务器端 state（R25）
- **给 advisor 的提示**：
  1. 强制 state vocabulary 约束（最高 ROI——直接掐死 `PROCESS_PAUSED`）
  2. 把"看到 client-side connection error 时优先验证 client-side jvm metric"做成显式 prompt 模板
  3. 把"cumulative counter 不增长 ≠ paused"做成显式约束（process paused 必须看 phase / restart）
