### 3. chaos 机制

HTTPResponseReplaceBody 在 ts-route-plan-service 收到 ts-travel2-service 返回的 GET `/api/v1/travel2service/routes/*` response 时，把 response body 替换为非法 JSON（开头是 token `nvpwzt`）。物理表现：

- ts-travel2-service 端正常返回 valid Response<Route> JSON
- ts-route-plan-service 收到 response 时 chaos sidecar 把 body 改成 `nvpwzt...`（非法 JSON）
- ts-route-plan-service 的 RestTemplate 用 Jackson 反序列化 → 抛 `JsonParseException: Unrecognized token 'nvpwzt'`
- caller (route-plan) 把这个异常包装成 RestClientException → SEVERE log（**1100 条同质**）→ HTTP 500 给上层
- caller 失败后会重试 → caller→callee 调用边 a_c=2223 vs n_c=451 (**4.929x**)，retry 风暴
- callee (travel2) 端 endpoint 完全正常（接收正常请求、返回正常 JSON），endpoint ratio 1.2-2.9x（只有轻微 retry 风暴打过来的 cascade）
- causal_graph 主 RC 标 `ts-route-plan-service`（caller 一端，因为它是直接报错的服务）；GT 字符串列入 ts-travel2-service 是 chaos 边的另一端

### 9. 失败池使用规则

```
本 case 命中标签：A2, A4, A8, A10, A11, B1
最有效组合：A11 + A4 + B1（错误信息因果归属反查 + log 字面/服务名相似陷阱反证 + caller-callee 边级 retry 风暴必查）
```

引用机制：
- **A11** 是起点：caller 报 JSON parse / Response<X> deserialization error 时必须把 X 解读为"caller 端反序列化对象类型"（不是"返回该类型的 service"）；caller 反序列化失败是 caller 端现象，应锚 caller 而非根据类型名找 service
- **A4** 是中间：log 里出现的 service 名 / 类型名都是 caller 报告对象，锚定前必须查该 service 自身 SEVERE / Error rate / trace count 直接证据；route-service 在 abnormal log 全 INFO + 0 SEVERE / 0 Error，自身证据不存在
- **B1** 是收尾：跑 caller-callee parent-child edge ratio 会立刻看到 route-plan→travel2 retry 风暴 4.929x（chaos 注入边）+ route-plan→route-service trace 数反而 0.127x（route-service 跟 chaos 边无关），从拓扑层挤掉 route-service 假说

### 10. 判断

- **数据是否完整**：✅ 完整（caller 1100 同质 SEVERE + caller-callee 4.929x retry 风暴 + server 端 endpoint baseline 全部齐全）
