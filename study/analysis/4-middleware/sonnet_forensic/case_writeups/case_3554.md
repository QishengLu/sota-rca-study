### 3. chaos 机制

HTTP-response-abort 跟 case 2830 的 HTTP-request-abort 区别：
- **request-abort**: 在 request 到达 callee **之前**拦截 → callee 完全 silent，caller-callee edge ratio ≈ 0
- **response-abort**（本 case）：request 到了 callee，callee **实际处理了请求**（产生 server span），response **回程**阶段被 abort → caller 看到 connection reset / no HTTP response，但 callee server span 看起来完成了

物理后果：
- **caller (ts-travel-service)**：outbound CLIENT span 大量 status_code=Error + 无 HTTP response（response 没收到）
- **callee (ts-basic-service)**：server span 完成（处理了请求）但 response 丢失 → **status_code 仍是 Unset/healthy**，**0 Error**（callee silence on errors 陷阱），但因 caller 重试少了 → trace count 整体 -48%
- **caller-callee edge ratio 0.634**（中度 drop，不像 request-abort 那样 0.003 极端）—— response-abort 的特征指纹

GT 共病的合理性：caller (travel) 看到 outbound failure + callee (basic) 流量被切但 silent on errors。**两端方向相反信号**（caller 满 error，callee 完全 silent）。

### 6. sonnet 完整推理路径

sonnet 跑 40 round → 锚 `ts-admin-user-service` HIGH_CPU "noisy neighbor on worker1" ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R3-5 | discovery + service-level error rate | 识别 travel-plan / route-plan / loadgen 是 cascade |
| 🟡 R7 | **看到 GT 直接证据但没识别** | "ts-travel-service: 81 errors on POST /trips/left (500), POST CLIENT span errors" — caller 端 outbound CLIENT span error，**HTTP-response-abort 指纹**，sonnet 看到没识别 |
| 🟡 R12 (msg=70) | **callee silence 没识别** | "ts-basic-service calls station/route/train/price - all with 0 errors" — callee silence on errors 是 response-abort 的标志，**sonnet 当作"basic-service 健康"反而排除它** |
| 🔴 R20 锚定瞬间 (msg=89) | metric drill | "**ts-admin-user-service: Abnormal avg=0.059, max=0.859 (almost 1 CPU!) — Normal: avg=0.005, max=0.008. ~11x avg, ~108x max difference**" — 用 ratio-only + max-only 锚定 |
| 🟡 R26 (msg=113) | **再次看到 GT 信号** | "ts-basic-service avg 29.71→80.80 (2.72x), ts-travel-service 也劣化" — GT 直接 latency 信号，**sonnet 仍合理化 ts-admin-user-service** |
| R30 (msg=129) | 加固 ratio-only | "ts-train-service byName max 24.8→2252 (90x)" — 被 worker1 同节点的"noisy-neighbor"假设吸引 |
| R33-34 (msg=135-141) | **worker-node 物理拓扑联想偏差**（跟 case 1140 同模式） | "**worker1**: ts-admin-user-service, ts-train-service, ts-travel-plan-service. **ts-admin 11x avg, 107x max — ANOMALOUS — 它在 worker1 starve 了其他服务**" |
| R37 (msg=149) | **host-level metric 误读** | "jvm.system.cpu.utilization shows worker1 0.05 → 0.44" — 系统级 CPU 是节点级 metric，sonnet 当 ts-admin 自身信号 |
| 🟡 R40 (msg=168) FINAL | 锁定（自相矛盾仍坚持） | "Root Cause: ts-admin-user-service (HIGH_CPU) - Normal avg 0.005 → Abnormal 0.059. Located on worker1, **does not appear in any failing traces**（自己已发现 trace=0！）, **noisy neighbor**" |

**sonnet 失败点（具体行为）**：
1. **看到 GT 教科书指纹两次但没识别**（R7 ts-travel POST CLIENT span 81 errors / R12 ts-basic-service callee silence on errors —— HTTP-response-abort 双端指纹都看到没识别）
2. **ratio-only fallacy（除以极小基数）**：用 ts-admin CPU avg 0.005→0.059 ratio 11x 当主证据，绝对值仍 < 0.1 cores，cpu_limit_utilization 仅 1.4%
3. **max-only fallacy**：用 max 0.008→0.859（107x）当判据，单点峰值不能代表持续负载
4. **锚定 trace=0 的无关服务**：**最严重失败**——ts-admin-user-service 在 normal AND abnormal 都 0 trace / 0 log，跟 chaos chain 完全脱节，sonnet 自己 R40 都承认 "does not appear in any failing traces" 仍坚持锚定
5. **silence-as-causation 倒因果**：ts-basic-service callee silence on errors（chaos response-abort 让 server span 看似 healthy）→ sonnet 当作"basic-service 健康"理由排除 GT
6. **worker-node 物理拓扑联想偏差**（跟 case 1140 同模式）：发现 ts-admin / ts-train / ts-travel-plan 都在 worker1 → 跳到 noisy-neighbor 假设
7. **host-level metric 误读为 pod-level**（跟 case 1140 同模式）：jvm.system.cpu.load_1m / jvm.system.cpu.utilization 是节点级 metric，被当 pod 自身信号
8. **没做 caller-callee edge ratio drop SQL**：(ts-travel → ts-basic) edge 0.634 + caller side outbound CLIENT span Error 是 response-abort 双端指纹
9. **anchor 锁死后过度合理化**：R20 锚定后 20+ round 全用在合理化，反复在 worker1 节点元数据找证据，强行建构 noisy-neighbor 因果链

- **共性 vs sonnet-specific**：**sonnet-specific failure** —— qwen3.5 命中 ts-basic-service ✓；sonnet 是 case 1140 + case 2830 失败模式的叠加版。**新发现的稳定失败模式：chain overlap 验证完全缺失**（锚定 trace=0 服务）。
- **给 advisor 的提示**：
  1. **chain overlap 必查**（锚定前强制确认服务在 abnormal_traces 里有数据）—— **新增最高优先级**
  2. HTTP-{request,response}-abort 双端指纹强规则（caller 端 CLIENT span no-response + callee 端 silence + count drop）
  3. ratio-only / max-only 反例校准（绝对值 + cpu_limit_utilization + 排名三重验证）
