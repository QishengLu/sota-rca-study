### 3. chaos 机制

HTTPResponseDelay 在 ts-travel-service client 端拦截 `POST /api/v1/basicservice/basic/travels` 的 **response**，给 server 返回的响应人为加 1462ms delay。物理上：

- ts-basic-service server **真实处理时间不变**（server span duration 仍接近 normal，因为 server 自己已处理完返回）
- ts-travel-service client 端 wallclock **多等 1462ms**（client span = server processing + 1462ms delay）
- ts-travel-service 调用 ts-basic 的 endpoint 整条链上 wallclock 都拉长，cascade 上游全部 latency 升高
- **副作用**：throughput 降低（每个请求等 delay → 单位时间处理少），ts-travel / ts-basic 双方 CPU **反向下降**

**关键 fingerprint** —— HTTPResponseDelay 跟其他 HTTPChaos 子类型的物理区别：
1. caller wallclock (client span) >> server processing time (server span)：本 case caller `POST /trips/left` avg 1584ms vs server `POST /travels` avg 61ms（差 ~1500ms ≈ chaos delay）
2. server endpoint count 不变（请求都到了 server）
3. caller endpoint count 减少（throughput 受限）
4. 双方 CPU avg 反向下降（处理慢 → 闲置）
5. server endpoint avg latency 反而下降（abnormal 期总流量减少，残余的快路径平均化）

GT 同时标 ts-travel-service（caller，wallclock 整体被 delay 加重）+ ts-basic-service（server，被加 delay 的目标）。

### 6. sonnet 完整推理路径

sonnet 跑 31 effective round → `ts-seat-service` HIGH_GC_PRESSURE/PROCESS_PAUSED ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R1-R5 | schema discovery + latency scan | 看到 ts-travel-plan-service / ts-route-plan-service / ts-travel-service avg 都升高 |
| R6-R10 | log query | R8 看 ts-route-plan SEVERE log "Connection reset on POST to ts-travel-service:8080/api/v1/travelservice/trips/left"——这条 log 实际指向 ts-travel-service 卡（chaos 让 travel→basic 慢 → travel→上游 timeout → 上游 connection reset），但 sonnet **没沿着这条线索 drill ts-travel** |
| 🔴 R12 锚定瞬间 | think | 看到 ts-seat-service `POST /seats/left_tickets` max **2314ms**（max-only！avg 34ms 接近 normal）→ "very high!" → 锚 ts-seat |
| R13-R15 | metric drill ts-seat | R14 看 ts-seat container.cpu max **0.994**（near 100%！）但 avg 0.218 vs normal 0.207 = 1.06x — sonnet 选 max 当判据 |
| R16-R20 | 时间序列 CPU | R17 看到 ts-seat CPU 14:04:53 = 0.994 → 14:05:03 = 0.019（断崖下降）→ sonnet 解为 "process resumed after pause" 加固 PROCESS_PAUSED 假设 |
| R21-R25 | hubble drops 验证 | 看 hubble drops 在 ts-seat 没特别异常，sonnet 用"network 侧 OK 因此是 ts-seat 自身 issue"合理化 |
| 🔴 R28 GC histogram | think | 看 jvm.gc.duration histogram → ts-seat 在 14:05:29 一次 GC pause **1.784s** → "**CRITICAL FINDING - JVM GC PAUSE ON ts-seat-service!**" → 加固 PROCESS_PAUSED |
| R29-R30 | rationalize | 用 GC 1.784s + CPU 0.994 spike + 时间断崖 编造 "ts-seat-service GC pause caused all upstream latency" 因果链 |
| R31 FINAL | output | `ts-seat-service` HIGH_GC_PRESSURE/PROCESS_PAUSED ❌ |

**sonnet 失败点**（去重独立化，每条独立可观测）：
1. **A3 反向 metric 信号被 max-only fallacy 覆盖**（R14 ts-seat container.cpu max 0.994 当判据，没看 avg 1.06x 完全 normal；distribution 全 normal 仅 max 偶发 spike → 当 sustained stress 误读）
2. **A11 因果链中段症状当机制**（R28 锚 GC pause 1.784s 当 RC 机制；GC pause 是高负载副作用，且 ts-seat avg 完全 normal 说明这次 GC 是偶发事件不是 chaos 引起）
3. **A10 schema 词表外的 state 自创**（PROCESS_PAUSED 不在词表 high_avg_latency / high_p99_latency / high_error_rate / timeout / injection_affected / healthy / unknown）
4. **A8 业务调用链外服务锚定**（ts-seat-service 跟 chaos 边 (travel→basic) 完全无关；ts-seat 不在 chaos `app_name` 也不在 `server_address`）
5. **A1 anchor 锁死后过度 rationalize**（R12 锚 ts-seat 后 18 round 全用在合理化：CPU 时间序列断崖 / GC histogram / hubble drops 都被解为 ts-seat 故障证据）
6. **A4 caller log 字面解读但走错方向**（R8 看到 SEVERE log "connection reset to ts-travel-service:8080/api/v1/travelservice/trips/left" — 这条 log 直接指向 ts-travel 是被卡的服务，但 sonnet 没把 travel-service 当 GT 候选，反而转去查 ts-seat）
7. **A9 多维度反证下仍锚**（ts-seat: avg CPU 1.06x ≈ normal + 端点 latency 0.84-1.08x ≈ normal + 不在 chaos 边 + service-level avg rank 不在前 18 + log 没指向它 — 5+ 维度反证仍锚）
