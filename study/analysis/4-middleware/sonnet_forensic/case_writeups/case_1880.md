### 3. chaos 机制

HTTPResponseReplaceBody 在 ts-food-service 的 client-side sidecar 拦截 `GET /api/v1/travelservice/routes/{tripId}` 的 **response body**，把 server 返回的合法 JSON 替换为 malformed body（含 token `lwwqt8`）。物理上：

- **server (ts-travel-service)** 实际 server-side 处理完全正常，返回的 response 是合法 Route JSON
- **chaos sidecar 在 food client 端**把 response body 替换为 malformed JSON
- **caller (ts-food-service)** Spring `RestClientException` 在 deserialize `Response<Route>` 时报 `Unrecognized token 'lwwqt8'`
- caller 自身 retry 触发 → minStopStations / cheapestRoute 类业务 endpoint 16x 重发 → 1380 SEVERE 累积

**关键 — 跟其他 HTTPChaos 子类型的物理区别**：
1. server 端 trace 看 server processing 时间正常（chaos 不让 server 慢）
2. server 端 trace 不报错（server 返回 200 OK + 合法 response，chaos 在传输路上替换）
3. caller 端 SEVERE log 是 `JsonParseException` / `RestClientException` 解析失败
4. caller→server edge 流量翻 6-12x（retry burden）
5. caller 自身 CPU 飙升（处理 retry + 解析失败）

GT 同时标 ts-food-service（caller，retry burden + 1380 JsonParseException）+ ts-travel-service（server，chaos 改 body 的目标）。

### 6. sonnet 完整推理路径

sonnet 跑 42 effective round → `ts-route-service` MALFORMED_JSON ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R1-R5 | schema discovery + error scan | 发现 ts-food-service 19.73% error rate (1381/6999 errors) |
| R6-R15 | log query | R10 看 SEVERE log: `RestClientException: Error while extracting response for type [Response<Route>] ... Unrecognized token 'lwwqt8'` |
| 🔴 R20 锚定瞬间 | think | "**malformed JSON parsing of `Response<Route>`** → 调 ts-route-service 拿 Route 实体失败" → 把 type 名 `Route` 字面解读为服务名 `ts-route-service` |
| R21-R30 | 验证 ts-route-service | R23 看 ts-route-service: HTTP 200 全部，logs 只有 INFO，metric 几乎 normal —— **这些是反向证据排除 ts-route-service**，但 sonnet 用 "返回 200 但 body 是 malformed" 合理化保留 anchor |
| R31-R40 | hubble + db.connection 对比 | R32 看 hubble traffic ts-route-service abnormal 是 normal 的 1.56x → "more requests" 加固 RC 假设（实际是 cascade 流量翻倍） |
| R41-R42 | rationalize | R41 看 trace 中 ts-route-service 在 food 失败 trace 里出现 200 OK，仍坚持 "返回 malformed body" |
| R42 FINAL | output | `ts-route-service` MALFORMED_JSON ❌ |

**sonnet 失败点**（去重独立化）：
1. **A4 caller log 字面解读 → 类型名 "Route" 误读为服务名 "ts-route-service"**（这是 case 675 connection reset 字面解读的更隐蔽变体——不是 URL 字面解读，而是 Java type 名字面解读）
2. **A8 业务调用链外（其实链上但跟 chaos 无关）服务锚定**（ts-route-service 在 cascade 链上但不是 chaos target，sonnet 没区分）
3. **A11 因果链中段症状当机制**（"ts-route-service 返回 malformed body" 是 sonnet 编造的机制，没物理证据；实际 ts-route-service 返回完全合法的 JSON）
4. **A9 多维度反证下仍锚**（ts-route-service: HTTP 200 全部 + INFO log + metric 几乎 normal + container.cpu **0.50x 反向下降** + 端点 ratio 0.94-1.21x normal — 5+ 维度反证仍坚持）
5. **A1 anchor 锁死后过度 rationalize**（R20 锚定后 22 round 用 hubble traffic / db.connection / trace 出现关系反复加固"ts-route-service malformed body"假设）
6. **没识别 chaos client-side 物理位置**（HTTPResponseReplaceBody 注入在 ts-food-service sidecar，不是 server 端；server 自己 trace 完全正常 = chaos 不在 server 上；sonnet 没把"server 端正常但 client 端 parse 失败"解读为 client-side chaos 注入指纹）
7. **没查 caller 自身 specific metric**（ts-food-service container.cpu 13.1x 飙升、jvm.cpu 14x 是 chaos 直接 caller-side 受害者证据；sonnet 全程没对 food 做 specific metric drill）

### 10. 判断

- **数据是否完整**：✅ trace/log/metric 都有，HTTPResponseReplaceBody 双 GT 指纹完整可见（caller CPU 13.1x、callee server endpoint 端 server processing 完全正常、edge ratio 11.7x retry、1380 JsonParseException incident-only）
- **chaos 是否生效**：✅ 完全生效（food→travel 调用 118→1380、1380 JsonParseException 全 incident-only、ts-food-service CPU 13x 飙升、loadgenerator timeout cascade 整链 ratio 8.9x）
- **真假盲区分类**：**(b) 模型盲区**（数据信号充足，sonnet+qwen3.5 共同被 (1) 命名混淆 (2) cascade 链反向 propagation (3) chaos client-side 物理位置不识别 三个陷阱误导）
- **失败模式归纳**：caller log Java type 名/URL path/服务名命名混淆 + cascade 链末端假设源头 + chaos client-side 物理位置不识别 + 反向 metric 反证未采纳 + anchor 锁死后 rationalize（共病 fingerprint）
- **共性 vs sonnet-specific**：**实际共病** — 两 model 同样锚 ts-route-service，task 标注为 sonnet-specific 是 stale 信息；本 case 暴露的是**两 model 共同盲点**（命名混淆 + cascade 反向 propagation + client-side chaos 不识别）
- **给 advisor 的提示**：
  1. SEVERE log 的 Java type 名（`Response<XEntity>`）不能直接当服务名；必须查 stack trace 后续行的实际 URL `http://X-service:port/Y`
  2. cascade 链 A→B→C 报错时，链末端 C 不一定是源头；必须查 C 自身的 specific metric (CPU / log) 是否真有异常
  3. caller 端 SEVERE log + caller CPU 飙升 + server 端完全正常（HTTP 200 + endpoint server processing normal）= client-side chaos 注入指纹（HTTPResponseReplaceBody / HTTPRequestReplaceMethod 等），caller + chaos `server_address` 才是 GT 双服务
  4. 候选服务 abnormal CPU < normal 0.5x 必须当反向证据从候选剔除
