### 3. chaos 机制

ChaosBlade NetworkLoss 在 pod 网络命名空间下用 tc qdisc 让 (travel-plan ↔ ui-dashboard) 之间 91% 的数据包随机 drop。极端 loss 率（91%）下应用层效果近似 NetworkPartition（基本通不过任何完整请求 — TCP retransmit 后仍有大概率丢失）。

预期因果链：
1. ui-dashboard 调 travel-plan 时 91% packet drop → TCP retransmit 风暴 + 大量请求 connect/read timeout
2. travel-plan 实际收到的 request 大幅减少 → travel-plan 自身处理量大幅下降 → **travel-plan trace count 几乎归零 + container.cpu 反向下降**（这是 NetworkLoss 极端版独家 signature）
3. ui-dashboard 端 endpoint wallclock 大幅升高（每次 request 等待 timeout）+ trace count 部分下降
4. 全集群 cascade（其他服务调 travel-plan 也失败 → 多边 ratio 接近 0）

NetworkLoss 91% 极端版 signature（比标准 NetworkPartition 更极端）：
- chaos source 端 (travel-plan) trace count -99%+（**几乎完全消失**，本 case 2451 → 6）
- chaos source 端 container.cpu **反向下降至 0.05-0.10x**（处理量几乎归零）
- chaos target 端 (ui-dashboard) endpoint wallclock 升 3-5x + count -67%
- chaos edge ratio < 0.01
