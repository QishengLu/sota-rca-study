### 3. chaos 物理机制 + 预期因果链

HTTPRequestDelay 的 chaos sidecar 在 **caller (ts-travel-plan)** pod 的 HTTP client 出站环节注入延迟——拦截发往 `ts-seat-service:8080/api/v1/seatservice/seats/left_tickets` 的请求，让 client 在发出前等待 2303ms。

**预期因果链**（按物理机制推）：

```
[chaos] sidecar 在 ts-travel-plan-service pod 出站层拦截发往 ts-seat-service 该端点的请求，sleep 2303ms 后才发出
   │
   ▼
[直接指纹] caller (ts-travel-plan) 看到的 client-span 时长 = chaos delay (2303ms) + server-side 实际处理时长
           ┌─ client-span: ts-travel-plan POST → ts-seat (≈ 2303 + 处理时长)
           └─ server-span: ts-seat POST /seats/left_tickets (处理时长本身，几乎不动)
           **client/server 时长差 ≈ delay_duration 是 HTTPRequestDelay 的因果指纹**
   │
   ▼
[二阶 cascade] ts-travel-plan endpoint POST /travelPlan/{quickest,minStation,cheapest}
              在内部串行调 ts-seat 的 left_tickets 多次 → endpoint 时长 = N × (2303ms + 处理) + 其他调用
              → ts-travel-plan endpoint avg/p95 暴涨
   │
   ▼
[SLO 末端] ts-ui-dashboard / loadgenerator 调 travel-plan 的 client-span 暴涨 → 部分超过 20s 客户端 timeout
```

GT = ts-travel-plan + ts-seat（chaos 注入点 + 受调对端，两端因果链最上游）。

### 4. 调用树（normal 期实测）

```
loadgenerator

          └─ POST → ts-seat-service /seats/left_tickets   ◀── chaos 注入这条 client 调用
          └─ POST → ts-route-plan-service / ts-station-service / ...
```

ts-config-service（sonnet 错锚）是被 ts-seat 内部调（用于配置查询），是 cascade 的下一层 victim。

#### 5.2 chaos 直接指纹：client/server span 时长不对称（HTTPRequestDelay 决定性证据）

**callee 端（ts-seat POST /api/v1/seatservice/seats/left_tickets server-span）**：

|---|---|---|---|---|

**因果指纹**：

### 6. sonnet 完整推理路径

sonnet 跑 ~17 effective rounds → final pred = `ts-config-service` HIGH_LATENCY ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R2-R3 | 看 SLO violation | 识别 ts-travel-plan endpoint POST /travelPlan/{quickest/minStation/cheapest} 几乎全部 timeout 20s |
| R4 | 看 SEVERE log | ts-travel-plan SEVERE 含 "ResourceAccessException... connection reset"—— 表明 ts-travel-plan 调下游被卡 |
| R5-R6 | 对比 ts-seat normal/abnormal trace | 看到 ts-seat POST /left_tickets normal 17.9ms vs abnormal 38.4ms（仅 2.1x）→ **判断 ts-seat 健康**（这是 client/server span 因果方向混淆） |
| R7-R8 | 看 ts-seat metrics | 看到 ts-seat container.cpu.usage max 0.915 + jvm.system.cpu.load_1m avg 48（实际 host-level，且 max 0.915 是 max-only） |
| R9-R10 | 看 jvm.system.cpu.load_1m 跨服务 | reflection 写："jvm.system.cpu.load_1m massively elevated for MANY services... ts-travel max 600.8, ts-route-plan max 600.8, ts-train max 600.8"——**已经发现"多服务数值完全一样"的 host-level 特征，但没识别**，反而当成"普遍 high CPU 压力" |
| R11-R13 | 找网络层 metric | dismiss network errors（k8s.pod.network.errors=0） |
| R14-R16 | 找特定 trace 深拆 | 找到 trace `6fe7da48...` 中 ts-seat::SeatController.getLeftTicketOfInterval 1.56s，内部 GET span 调 ts-config-service::ConfigRepository.findByName 1.52s |

| R18 final | 写答案 | ts-config-service HIGH_LATENCY |

**sonnet 失败点**：

1. R5-R6 看到 ts-seat server-side span avg 仅 2.1x → 判 ts-seat 健康，没意识到 HTTPRequestDelay 让 server-side span 几乎不动是**预期**（chaos 在 caller 端注入）
2. R9-R10 看到 jvm.system.cpu.load_1m 多 service 数值完全相同（ts-travel/ts-route-plan/ts-train 全是 221.86），但当成"普遍 high CPU"，没识别 host-level vs pod-level
3. R17 用单个 trace 内 ts-config GET max 1.52s 锚定，没回头看 ts-config service avg 仅 2.4x（跟 ts-seat 2.2x、ts-user 3.4x、ts-route 2.9x 一样的普遍 cascade 升高）
4. 没把 chaos 物理机制（HTTPRequestDelay/ResponseDelay/Network 类）摆上候选；69 round trajectory 中没出现"chaos 注入点 caller-side"的因果方向假设
5. 没做 client-span vs server-span 时长不对称比对——这是 HTTPRequestDelay 的因果指纹
