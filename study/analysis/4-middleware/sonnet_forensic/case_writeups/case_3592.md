### 3. chaos 机制

HTTP-response-delay 的物理机制（区别于 case 2830 request-abort 和 case 3554 response-abort）：
- request 正常到达 callee（ts-travel2-service），callee 实际处理了请求，server span 完成
- 在 response **回程**阶段被注入 4548ms delay（sidecar / iptables 拦截 response 包 4.5 秒后才放行）
- caller (ts-route-plan-service) 阻塞等待 response 4.5 秒 → caller 自己 endpoint wallclock 高（包含 delay）
- chaos 持续 4 秒，每个 affected request 等 4.5 秒——chaos window 内只能完成几个 request

物理机制 → 预期因果链信号 prior（**这是因果归因的起点**）：
1. **caller 端 endpoint wallclock 数值 ≈ delay_duration（4548ms）** —— 这是 HTTP-response-delay 的**决定性数值指纹**
2. **caller 端 outbound CLIENT span（指向 callee）avg 高**（含 delay 等待）
3. **callee 端 server span avg 不变**（callee 自己处理时间不变）
4. **caller-callee child span 数减半**（caller 重试少了）
5. cascade 上游（travel-plan / ui-dashboard）也卡在等 caller
6. 跟 chaos 边无关的服务（如 ts-travel-service）的流量 / CPU 会 cascade 下降——这是 **victim 现象，不是 RC**

GT 共病合理性：chaos 注入点 `app_name=ts-route-plan-service` + `server_address=ts-travel2-service`，注入参数把两端都标记为 GT。

| ts-route-plan-service `POST /minStopStations` p50 | — | 4777.9 ms | p50 也跟 delay 数值接近 |

**因果链验证 ✅**：caller endpoint avg **几乎完全等于 chaos delay_duration**——这就是 HTTP-response-delay 的物理机制留下的"签名"。

#### 误锚 service 实情（sonnet & qwen3.5 都锚的 ts-travel-service）

| 维度 | normal | abnormal | 解读 |
|---|---|---|---|
| trace count | 3011 | 1825 (-39%) | **流量减半（cascade 切流量 victim 现象）** |
| trace avg duration | 42.3 ms | **25.3 ms（反而下降）** | 不是 RC（RC 不会让自己 avg 下降） |
| status_code Error | 0 | **0**（全 Unset/healthy） | **silence on errors**——无 error log，无 error span |
| log levels | INFO 2771 | INFO 1721 | 无 ERROR/SEVERE，纯 INFO 减半 |
| `k8s.container.restarts` | 0 | **0** | **PROCESS_PAUSED 反证 1**：没重启 |
| `container.memory.working_set` | 759 MB | 788 MB | **PROCESS_PAUSED 反证 2**：几乎不变（pause 应有异常） |
| `container.cpu.usage` avg | 0.250 | 0.218 | CPU 下降（cascade 切流量造成，**因为没活儿干**而不是 paused） |

**因果方向辨析 ✅**：ts-travel-service 的"trace count -39% + CPU 下降 + status_code 全 healthy"链条因果方向是**被动**的：
> 上游（route-plan）卡在等 travel2 response 4.5s → route-plan 来不及发出对 travel-service 的下游调用 → travel-service 收到的 request 减少 → travel-service CPU 下降 + trace count 减半

这是 cascade 切流量的 **victim 现象**，因果链方向是 "上游卡 → 下游空闲"。
PROCESS_PAUSED 假设要求的因果方向是 "下游 paused → 上游受影响"——跟实测因果方向**完全相反**。

### 6. sonnet 完整推理路径

sonnet 跑 36 round → 锚 `ts-travel-service` PROCESS_PAUSED ❌

| Round | 行为 | 关键决策 |

| 🟡 R10 (msg=27) | **看到决定性数值指纹但没识别** | "TravelPlanController.getByMinStation avg **5178ms**" + R12 "RoutePlanController.getMinStopStations avg **4979ms**" —— 这两个数字几乎完全等于 chaos `delay_duration=4548ms`，但 sonnet 没问"这个 4.9s 跟 fault_type=8 HTTP-response-delay 的 delay_duration 对应吗？" |
| R14 | 否定网络层 | "No network errors in any service" — 但 chaos 是 sidecar/iptables 层不会出现在 hubble drop counter，否定逻辑错 |

**sonnet 整体推理偏离**：从 R10 看到 caller endpoint avg 4979ms ≈ delay_duration 4548ms（决定性数值指纹）开始，**没有一次回到 chaos 物理机制层校核因果链**。R26 自己注意到"travel-service all Unset/OK status puzzling"（victim 现象的因果反常）也没触发回退，反而沿 victim 信号链路深挖，最终把 cascade 切流量造成的 CPU 下降锚定为主动 PROCESS_PAUSED。

### 10. 判断

- **数据是否完整**：✅（caller wallclock ≈ delay_duration 决定性数值指纹 + caller-callee 双面不对称 14.2x + PROCESS_PAUSED 物理反证 restart=0/memory.working_set 不变 都齐全）
- **chaos 是否生效**：✅（caller endpoint avg 4901.6ms ≈ delay_duration 4548ms，物理机制完整生效）
- **真假盲区分类**：**(b) 框架/模型盲区** —— 数据有大量直接 chaos 物理机制层指纹，augmented_question 元数据也明确给出 chaos 类型 + 关键参数，但 sonnet 没把元数据当因果归因 prior，全程统计驱动归因
- **失败模式归纳**：A（chaos 物理机制层先验缺失）+ B（victim 现象当 RC + 假说没经物理机制层校核）+ C（caller-callee 双面信号不对称查询缺失）
- **共性 vs sonnet-specific**：**A + B + C 都是两 model 共偏离**（qwen3.5 锚同一个 victim 服务，本质同样是缺 chaos 物理机制层先验 + 没做因果方向辨析）；**sonnet 特有**仅在表面行为上（用 PROCESS_PAUSED 标签 + 多 round 沿 victim 信号链路深挖加固假说），根因因果归因偏离两 model 一致
- **给 sonnet 中间件设计的提示**：A + C + B 三条组合（详见 Section 9）。把 augmented_question 里的 chaos 元数据从背景信息升级为归因起点；用双面不对称 SQL 提取物理机制层指纹；用因果方向辨析 + 物理反证阻止把 cascade victim 当 RC
