### 3. chaos 机制

JVMException 通过 ChaosBlade 字节码增强 `PreserveServiceImpl.preserve`：每次调用直接抛 `java.lang.RuntimeException("Unexpected error")`。注入位点是 preserve 方法的入口，因此 chaos 触发时 preserve 在做任何下游调用之前就先失败。

因果链：UI 调 `POST /preserveservice/preserve` → controller 调 service.preserve → 该方法**入口处直接抛 RuntimeException**（chaos 注入）→ 9 个下游服务（security, contacts, basic, seat, order, travel, food, user, assurance）**全部不被调用**（因为 preserve 在 entrypoint 就 fail 了）→ Spring DispatcherServlet 捕获并记 SEVERE log → 返回 HTTP 500。

GT 单服务：`ts-preserve-service`。

### 6. sonnet 完整推理路径

sonnet 跑 79 msg（32 round）→ `rabbitmq` ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R3-7 | 查 abnormal preserve endpoint error | **🟢 正确锚定**：ts-preserve-service 100% Error (112/112) |
| R10 think | reflection | 显式说 "**when errors happen, there are NO downstream calls being made by ts-preserve-service**" — 即认知到入口处抛异常的 JVMException 签名，但未把 ts-preserve-service sticky |
| R12-15 | 查 RuntimeException + preserve metric | 发现 SEVERE: "RuntimeException: Unexpected error"；container metric 正常；think "exception logs don't provide more detail" → 觉得证据不够，开始 broaden |
| R20 | 全 cluster scan ERROR/SEVERE log count | 看到 ts-delivery-service 48 ERROR + ts-notification-service 48 ERROR |
| R22 think | reflection | "ts-delivery-service errors..." → 把 delivery/notification 错误数当 chaos 信号（未跑 chronic check）|
| R24 | 看 delivery/notification log | 看到 "Failed to check/redeclare auto-delete queue(s)." + "Restarting Consumer" + "Attempting to connect to: [ts-rabbitmq:5672]" |
| 🔴 R25 锚定瞬间 think | "**CRITICAL FINDING! Both ts-delivery-service and ts-notification-service are failing with 'Failed to check/redeclare auto-delete queue(s).' This is a RabbitMQ error!**" |
| R28-29 | 查 hubble_drop_total + rabbitmq metric | 看到 hubble drop 3 packets from rabbitmq-0 → 解读为"network drops from rabbitmq" |
| R30 think | reflection | "Hubble shows packet drops from ts5/rabbitmq-0 starting at 15:11:53..." → 推理为 rabbitmq network partition |
| R31 verify normal preserve log | reflection | 发现 normal 期 preserve ERROR 是 "[Step 4][Do Order][Create Order Fail]"（业务级），abnormal 是 "RuntimeException Unexpected error" → 注意到差异但**不回到 preserve 假说** |
| 🔴 R32 final | think | 输出 RC: `rabbitmq` + state=['NETWORK_PARTITION']（实测 chronic + 3 packet drops + rabbitmq broker 健康 + preserve 不经过 rabbitmq）|

**sonnet 失败点**：
1. R10：正确认知 entry-fail 模式但未把 ts-preserve-service sticky
2. R22：把 delivery/notification 错误日志当 chaos 信号，没做 normal vs abnormal chronic 检查
3. R25：基于 chronic noise 形成 rabbitmq 假说
4. R28-30：3 packet hubble drops 被字面解读为"network partition"，量级未对照 noise floor
5. R32：输出非合法 service 名 `rabbitmq` + 自创 state `NETWORK_PARTITION`，且 preserve 跟 rabbitmq 无任何调用关系（因果链断裂）

- **数据是否完整**：✅ 112 SEVERE RuntimeException + 9 条 preserve 下游 edge ratio=0 = JVMException entrypoint-fail 教科书指纹
