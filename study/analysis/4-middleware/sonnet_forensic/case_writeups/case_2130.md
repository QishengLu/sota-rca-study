### 3. chaos 机制

ChaosBlade JVM agent (ByteBuddy) attach 到 ts-station-service JVM，把 `fdse.microservice.StationApplication.main` 的返回值改写为 0。**问题**：`main` 方法是 Spring Boot 启动入口，应用启动后 main 已经返回；运行期 patch 它的 byte code 不会有业务效果（main 不会被再次调用）。

**实测**：chaos 几乎未生效。唯一可观测到的是 ByteBuddy agent attach 引发的 byte code redefinition 副作用（station-service 的 JVM 在 redefinition 期间短时 CPU 升 + OTel SDK / hubble eBPF 时序短暂错乱）。归类 `unsolvable:chaos-side-effect-only`，与 case 339 / 572 / 3114 同类。

**sonnet 失败点**：
1. **A3 max-only fallacy**：用 SELECT route max=106s 单点 outlier 锚定 RC，没看 distribution（p50=1.0ms 完全等于 normal、p95=2.0ms 仅 2x）。202 个 abnormal trace 中只 2 个超长 outlier — 这是单点事件，不是稳态故障
2. **A11 错误信息字面解读**：把 mysql page_cleaner "26071ms loop" log（仅 1 行）当 "DB I/O starvation" 直接因果，没做 chronic check（normal=0 abnormal=1，1 行 log 不构成稳态故障证据），也没问"为什么这 1 行 log + 那 2 个 SELECT 106s outlier 同时刻发生"
3. **A9 cascade 末端锚定**：route-service 在调用树末端，是被调方；它的下游 mysql 也只在 1 个时刻有 1 行 page_cleaner log。sonnet 没问"如果 route 是 origin，为什么 99% 的请求 SELECT route 仍只用 1ms"
4. **CPU 三维度方向矛盾未识别**：container.cpu.usage max 9.16 vs jvm.cpu.recent_utilization 0.82x 下降 — 应触发 dismiss "持续 cpu 烧" 假说
5. **B9 baseline cross-rank 不彻底**：sonnet 没扫 station-service（GT），station endpoint 2.95x + jvm.system.cpu.utilization 2.82x + hubble HTTP NaN 是弱但齐全的 chaos 副作用指纹
6. **chaos-effectiveness gate 未识别**：全表 ratio_count 全部 0.4-0.47（系统总流量减半，不是某 service 特异下降）+ ratio_avg 极端值集中在 cascade 末端 + GT 在 noise floor — 这种 pattern 提示 chaos 主链路未生效，应降级结论可信度

### 10. 判断

- **数据是否完整**：⚠️（abnormal 数据完整，但 chaos 主信号物理上不存在；有 byte code redefinition 副作用）
- **chaos 是否生效**：❌ **几乎未生效**。chaos target `main` 方法在运行期不被调用；唯一观测到的是 ByteBuddy attach 副作用（station jvm.cpu 2.82x + hubble NaN）。SELECT route 106s 是 cascade 末端单点 outlier（202 trace 中仅 2 个），不是稳态故障
- **真假盲区分类**：(c) chaos 未生效（与 case 339 / 572 / 3114 同类）— 但**不是完全 unsolvable**：理论上"GT 弱副作用三件套（jvm.cpu 2.82x + hubble NaN + endpoint 2-3x）+ chaos-effectiveness gate 降级"可命中 station-service，只是信号比 cascade 末端 outlier 弱 100x
- **失败模式归纳**：A3（max-only fallacy）+ A11（mysql 1 行 log 字面当稳态 DB I/O 故障）+ A9（cascade 末端锚定，无下游/位置反证）+ A12（CPU 三维度方向矛盾未否决）+ B9（baseline cross-rank 不彻底，未扫 GT 弱信号）+ 新模式 chaos-effectiveness gate
- **共性 vs sonnet-specific**：
  - **完全共性**：sonnet 跟 qwen3.5 都被 cascade 末端 SELECT route 106s 单点 outlier 锚走，都没识别 chaos-on-main 几乎未生效 + GT 弱副作用三件套
  - **sonnet-specific**：sonnet 多构造了 mysql page_cleaner log 因果链（A11 字面解读 + 编织 DB-starvation 故事），把弱信号"DB I/O starvation"假说包装得更"完整"反而更难拆。qwen3.5 没碰 mysql 这条线
- **给 sonnet 中间件设计的提示**：A3 + A12 + B9 三条组合（详见 Section 9）。核心：**distribution 强制（p50/p95 与 normal 比 + outlier ratio）+ CPU 三维度方向反证（container vs jvm.system vs jvm.cpu.recent）**应作为锚定前的硬关卡，单点 outlier + 单行 log 不构成稳态 RC 证据

### 11. causal_graph 正确性检验

⚠️ causal_graph 只列了 root_cause = `service|ts-station-service` + state ["unknown"]，没有任何 span 节点的 state 标注（dump 里只有 root_causes）—— 这跟 chaos-on-main 几乎未生效一致：图模型也没观测到稳态 propagation。

实测验证：station-service 端点确实呈现 2-3x 弱上升 + jvm.cpu 2.82x + hubble NaN，跟"chaos 副作用"假设一致；GT 在 service-level 没有强信号是数据真实状态。本 case 应归到 `unsolvable:chaos-side-effect-only`——任何归因都依赖能否识别 chaos 未生效 gate，sonnet 没识别此 gate 是核心失败点而非细节失败。
