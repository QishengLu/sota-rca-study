### 3. chaos 机制

HTTPRequestAbort 在 ts-route-plan-service 客户端**精确**拦截 `POST /api/v1/travel2service/trips/routes` 这条边的请求，直接 abort。物理上：

- **caller (ts-route-plan-service)** 调 travel2/trips/routes 全部 `Connection reset / I/O error`，触发 caller 自身的 retry 逻辑 → minStopStations endpoint 16.5x 重发
- **server (ts-travel2-service)** 实际收到的 `POST /trips/routes` 请求**反而减少**（被 abort 在 caller 端），CPU/JVM 占用反而下降到 0.31x
- **副作用 — 业务并发的"连带流量放大"**：trainticket 业务上 ts-route-plan-service 的 minStopStations / cheapestRoute 算法**本来就同时双调** ts-travel-service 和 ts-travel2-service（dual-database 校验模式：travel 表查 trip + travel2 表查 trip2）。所以每次 retry 整个 minStopStations endpoint 时，travel 那一边也被一起放大调用 16.5x —— 但**travel 那边的调用是成功的**（不是 connection reset），只是流量增加导致偶发 GC pressure / max CPU spike

**关键**：travel 跟 travel2 的"同时被 SEVERE log 提到"不是因果连带（不是 fallback），而是**业务并发双调 + retry 把整个端点放大**的副作用。SEVERE log dedup by URL 显示：`travel2/trips/routes` 1188 次（chaos 精确生效），`travel/trips/left` 仅 1 次（偶发噪声）—— 1188 vs 1 的悬殊比例本身就是 chaos 精确打 travel2 的强证据。

GT 同时标 ts-route-plan-service（caller，retry burden 直接受害）+ ts-travel2-service（被精确切断调用的 server），因 chaos 切的是这条 caller→callee 边。

### 6. sonnet 完整推理路径

sonnet 跑 47 effective round → `ts-travel-service` HIGH_GC_PRESSURE/CONNECTION_RESET ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R1-R5 | schema discovery | 摸表结构 |
| R6-R10 | error span scan | 发现 ts-route-plan-service `POST /minStopStations` 1188/1188 errors (100%) |
| R11-R15 | SEVERE log query | 拉 ts-route-plan-service SEVERE log，看到 "Connection reset" 调 ts-travel-service 和 ts-travel2-service 的 log template |
| R16-R20 | log 内容分析 | 解读 4 类不同的 connection reset error 来源 |
| R21-R25 | metric 对比 ts-travel-service vs ts-travel2-service | 看 CPU/memory 对比，发现 ts-travel-service max CPU 0.96，ts-travel2-service avg 反而下降 |
| R26-R30 | hubble HTTP duration NaN | 发现两 service hubble HTTP duration 都 NaN abnormal → 当 "network connectivity issue" 解读 |
| R31-R35 | JVM GC histogram | **R32 抓到 ts-travel-service GC max 2.612s vs normal 0.425s (6x！)** |
| 🔴 R36-R40 锚定瞬间 | think + hubble traffic | 看到 ts-route-plan→ts-travel-service traffic 10168 vs normal 4531 (2.24x more) → think_tool 写: "ts-travel-service 是被打高负载的服务，导致 GC pressure → connection reset" |
| R41-R45 | 加固 anchor | 查 queueSize / 时间序列 CPU peak / hubble 详细对比，反复合理化 ts-travel-service |
| R47 FINAL | output | `ts-travel-service` HIGH_GC_PRESSURE/CONNECTION_RESET ❌ |

**sonnet 失败点**：
1. **SEVERE log 不按 URL count 排序**（log dedup 显示 1188 (travel2/trips/routes) vs 1 (travel/trips/left) vs 1 (travel2/other) 的悬殊比例；sonnet 按 distinct template 排序看到"4 类不同 connection reset"就停，没看 count distribution）
2. **业务并发 vs fallback 误读**（ts-travel-service 的 retry 流量 16.5x 是业务上 minStopStations 算法本来就双调 travel+travel2 的副作用，不是 fallback；sonnet 把 travel-service 流量增多解读为"travel 是受害最严重的服务因此是 RC"）
3. **服务名相似性陷阱**（ts-travel-service vs ts-travel2-service 一字之差；sonnet 选了"看起来更主流"的 ts-travel-service，没把 chaos `server_address` 字段精度当 GT 候选首选）
4. **max-only fallacy**（ts-travel-service container.cpu.usage max 0.96 当主证据，没看 avg 0.166 比 normal 0.192 还低 0.86x — distribution 完全 normal，仅 max 偶发 spike）
5. **cascade victim 锚定**（ts-travel-service GC pause 6x + traffic 2.17x 都是 retry 流量副作用——业务双调 + caller retry 16.5x 让 travel 一起被放大；sonnet 把这个 cascade 解读为 RC）
6. **没查 ts-route-plan-service 自身 CPU/JVM**（ts-route-plan jvm.system.cpu.load_1m 33x、container.cpu.usage 7.9x 是 chaos 直接受害者最强证据；sonnet 全程没对 ts-route-plan 做 specific metric drill）
7. **没读 SEVERE log 的 server_address + route 路径精度**（log 1188 次明写 `Connection reset to ts-travel2-service:8080/api/v1/travel2service/trips/routes` ——这就是 chaos `server_address` + `route` 字段的精确复刻；sonnet 没用这个 1188 vs 1 的悬殊去筛选）
8. **silence-as-causation 的反向版**（ts-travel2-service avg CPU 0.31x 反向下降是 chaos abort 的指纹——请求没到 server；sonnet 把它当"travel2 没事"反而排除候选）
9. **anchor 锁死后过度 rationalize**（R32 锚 GC 6x 后剩 15 round 全在用 hubble traffic / queueSize / CPU peak time series 加固 ts-travel-service，没回头质疑 anchor）

### 10. 判断

- **数据是否完整**：✅ trace/log/metric 都有，HTTPRequestAbort 双 GT 指纹完整可见（caller CPU 33x、callee CPU 0.31x 反向下降、edge ratio 0.171/2.169 双向）
- **chaos 是否生效**：✅ 完全生效（ts-route-plan→ts-travel2 调用 ratio 0.171 切掉 83%、ts-route-plan retry burden 1188 SEVERE log + CPU 33x、loadgenerator 20s timeout cascade 整链 ratio 7-14x）
- **真假盲区分类**：**(b) 模型盲区**（数据信号充足，sonnet 自己被 metric 显眼度 + 服务名相似 + max-only 三连陷阱误导）
- **失败模式归纳**：服务名相似性陷阱 + max-only fallacy + cascade victim 锚定 + 反向 CPU 信号未识别 + anchor 锁死后 rationalize + 没读 server_address log 精度
- **共性 vs sonnet-specific**：**sonnet-specific failure**（qwen3.5 走 error count 路径绕开了陷阱）；其中 anchor 锁死后 rationalize 是 sonnet 跨 case (4423 + 2011) 重复 fingerprint
- **给 advisor 的提示**：
  1. caller error log 里 `Connection reset to http://X:port/Y` 的 X 必须当 GT 候选首选（不让旁路相似名服务并列竞争）
  2. abnormal CPU < normal 是 HTTPRequestAbort 类 chaos 的反向指纹（callee 没收到请求）
  3. caller-callee edge ratio 升序+降序双扫，识别 chaos target edge 跟 fallback 旁路 edge
  4. metric max 高但 avg/p95 接近 normal 不能当 RC 主证据
