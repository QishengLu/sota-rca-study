### 3. chaos 机制

ContainerKill 向 ts-food-service 容器发送 kill -9，进程终止后 K8s 立即重启容器，container.restarts 0→1（abnormal 期 pod `ts-food-service-55f49f6b59-99lgr`）。重启期间约 1 分钟无法处理请求：trace count 降至正常期 45%（2245→1016），CPU 在 JVM 重启预热飙升 7.54×（avg 0.096→0.725），cpu_limit_utilization 6.37× ratio。下游传播：ts-preserve-service 调 ts-food-service POST /foodservice/orders 时连接受阻，p99 从 736ms 升至 6913ms（≈9×）。

### 6. sonnet 完整推理路径

sonnet 跑 28 round → 最终 response 锚 `ts-rabbitmq` DNS unreachable ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R1–R3 | list_tables + schema + plan | 计划：foodservice 和 preserveservice 两条 SLO |
| R4–R6 | 查 trace error + ratio | "No ERROR status codes...issue is HIGH LATENCY"；list 高延迟服务 |
| R7–R8 | 端点 distribution normal vs abnormal | preserve p99 736→6913 (9×)，food p99 256→1544 (6×) |
| R9–R10 | 查 ERROR logs by service | 看到 ts-notification + ts-delivery `Failed to check/redeclare` + ts-food `[getAllFood] Get Food Request Failed!` |
| 🔴 R10 | think | "**RabbitMQ connectivity issue**: ts-notification + ts-delivery + ts-food all failing on rabbitmq" → **早期锚定 RabbitMQ** |
| R11–R14 | 查 slow trace 调用链 | 顺着 preserve→travel→seat→basic 找下游延迟，不再回头查 ts-food-service 自身 |
| R15–R16 | 查 ts-basic-service 数据 | 发现 ts-basic-service `BasicController.queryForTravel` 2878ms，认为是 cascade |
| R17–R18 | 查 rabbitmq pod metrics | "rabbitmq-0 pod RUNNING with low CPU, **no restarts**" → 解读为 "DNS issue not pod crash" |
| R19–R22 | ts-basic-service CPU 时序 | 发现 ts-basic-service CPU 时序中 0.636 峰值，认为 cascade |
| R23–R26 | 查 ts-order-service 端点 | p99 5→45ms，max 2944ms outlier，归入 cascade |
| R27 | 查 RabbitMQ restart 验证 | 确认 rabbitmq-0 restart=0 |
| 🔴 R28 | think final | "RabbitMQ restarts: 0…confirms the issue is NOT the pod being killed…**DNS resolution failure for ts-rabbitmq**" |
| **FINAL** | response | "Root Cause: **`rabbitmq` (Kubernetes Service: `ts-rabbitmq`)** unreachable via DNS" |

**sonnet 失败点**：
1. R10 看到 rabbitmq DNS error 即锚定，未做 normal vs abnormal 对比 SQL（实测 normal=418 vs abnormal=312）
2. R17、R27 sonnet **只查了 rabbitmq 自己的 restart counter（=0）**，没运行全局 restart 差分扫描；如果跑了 `WHERE metric='k8s.container.restarts' AND value > 0` 全局 GROUP BY pod，会立即看到 `ts-food-service-55f49f6b59-99lgr` 全局唯一新增 restart=1
3. R11–R26 整个调查转向 cascade 下游（basic / travel / seat / order）找延迟来源，**始终没有回头查 ts-food-service 自身的 specific metric value（restart / CPU ratio）**

### 9. 失败池使用规则

```
本 case 命中标签：A5, B3
最有效组合：A5 + B3（chronic check 强制 + specific metric 显式 list 查 value，包含全局 restart counter 差分扫描）
```

A5 处理方式：锚定 ERROR pattern 前必须运行 normal vs abnormal 对比 SQL；ratio < 2 或 normal ≥ abnormal → dismiss。

B3 处理方式：怀疑 service 时强制按显式 metric list 查 value（不是查 metric 名字 enumerate）；额外强化：当无明显候选时强制运行**全局 restart counter 差分扫描**（abnormal max - normal max > 0 的 pod），避免只在已成假说服务上做 verification。

### 10. 判断

- **数据是否完整**：✅ restart=1、CPU 7.54×、trace count 0.45× 全部可查
- **chaos 是否生效**：✅ restart 0→1 + CPU avg 7.54× + trace 数降一半
- **真假盲区分类**：**(b) 框架/模型盲区** — 数据有完整 ContainerKill 指纹（全局唯一 restart=1 + CPU 7.54×），sonnet 在调查 28 round 全程未运行任何与 ts-food-service 自身 specific metric value 相关的 SQL，转而被 chronic rabbitmq 配置 noise 吸引，并仅在已成假说上做 confirmation
- **失败模式归纳**：A5（chronic noise 锚定，未做 normal vs abnormal 对比；R28 verification 只查 rabbitmq 自己 restart=0 而未做全局差分）+ B3（ts-food-service specific metric value 整轮未查；缺全局 restart counter 差分扫描）
- **共性 vs sonnet-specific**：sonnet-specific（qwen3.5 正确）；本 case 与 case 1144 失败模式 95% 重合（同 chaos 类型 ContainerKill on ts-food-service，同锚 chronic rabbitmq DNS）→ sonnet 在 ContainerKill on 具有 chronic rabbitmq 客户端配置的服务上有**稳定可复现**的失败模式
- **给 sonnet 中间件设计的提示**：A5 + B3 两条组合（同 1144），重点强化 B3 中"全局 restart counter 差分扫描"以应对"无明显 service-level ratio 候选"的 case（本 case ts-food-service 在 SQL 1 排第 3，不是首选）
