### 3. chaos 机制

ChaosBlade NetworkDelay 在 pod 网络命名空间下用 tc qdisc 给 (seat ↔ travel-plan) 之间的数据包加 1456ms ± 280ms jitter。注意业务调用拓扑实际方向：**ts-travel-plan-service → ts-seat-service**（travel-plan 是 caller，seat 是 callee；chaos 配置内部 direction='to' 不等于业务调用方向）。

预期因果链：
1. travel-plan 调 seat 的 outbound HTTP CLIENT span 每次延迟 1456ms（callee server-side span 时长不变 + 网络传输额外 1456ms）
2. travel-plan endpoint 总 wallclock 大幅升高（每次 RPC 加 ≈ 1456ms）—— **travel-plan endpoint avg 33.8x 是最强 chaos signature**
3. travel-plan 端 thread 阻塞等待 seat 响应 → caller-chain 上游 (ui-dashboard / loadgen) 跟着慢 6-10x
4. seat 端被动接收 throughput 下降 + 自身 jvm.cpu 升高（13.21x）—— retry/timeout 风暴
5. 全集群 cascade：所有调 travel-plan / seat 的服务 thread 跟着阻塞，jvm.system.cpu.load_1m 普遍暴涨 13-17x

NetworkDelay 标志 signature（按 skill 库）：edge ratio 中度降（0.3-0.7）+ hubble drop counter 升高 + **caller wallclock 与 callee server-side 时长不对称**（差值 ≈ delay_duration）。

### 9. 中间件提示维度（针对本 case 4 条失败模式 1:1 设计）

| 失败模式 | 中间件提示维度 | 触发时机 |
|---|---|---|
| A2 + A11 | **GT 顶点信号 + chaos 物理机制 prior**（强化版）：当 reflection 中识别到某 service avg ratio 在全集群 top1 且 ≥ 10x 时，强制按 chaos 物理机制 prior 解读——不能默认 "endpoint 慢一定因为下游慢"。问题清单：(1) 增加的 wallclock 量级是否对应某个 chaos 类型的 delay_duration（NetworkDelay 1000-3000ms / NetworkPartition 30-130s timeout / JVMStress GC pause 量级）；(2) 该 endpoint 的 callee server-side avg 是否同样大幅升高（同 → callee 自身故障 / 不同 → caller-side 网络 chaos）；(3) chaos 物理机制能否解释全集群 cascade 模式（NetworkDelay 一条边 1456ms 让全集群 thread 阻塞 jvm.cpu 13-17x；JVM stress 单服务故障不会让全集群 cascade）。本 case travel-plan 33.8x ≈ 2.3 × 1456ms 直接命中 NetworkDelay；callee server avg 1.71x ≪ 33.8x 排除 callee 自身故障 | reflection 看到 service avg ratio top1 ≥ 10x 时 |
| A3 + 子表现 | **max-only outlier 反证 + 样本数门槛**：用 metric max 作 RC 证据时强制：(1) 样本数门槛——histogram records ≥ 5 才能用 max；< 5 → 单点事件不能当 RC；(2) 配套维度反证——GC long pause 候选必须配合 container.cpu 升 + jvm.cpu 升 + endpoint avg ratio 升组合，任一反向（如本 case route-service container.cpu 反而下降 0.49x）→ dismiss；(3) 系统性 vs 单点解释——chaos 系统性现象（全集群 wallclock 升）不能用单点事件（一次 GC 3s）解释 | 用 metric max 作 RC 证据时 |

### 10. 判断

- **数据是否完整**：✅ 数据齐全，NetworkDelay 经典 signature（travel-plan endpoint 33.8x ≈ 2.3 × 1456ms delay + caller-callee 不对称比 19.8x + seat jvm.cpu 13.21x）
- **chaos 是否生效**：✅ 完全生效（chaos delay 1456ms 物理机制直接对应 travel-plan wallclock 增加 3398 ms）
- **真假盲区分类**：(b) 框架/模型盲区——sonnet 实际看到 chaos GT 顶点信号（travel-plan 33.8x）但因 chaos 物理机制 prior 缺失被解读为"下游慢"
- **失败模式归纳**：A2+A11（GT 顶点未锚 + chaos 物理机制 prior 缺失，跟 1495/4229 同病灶）+ A3 子表现（2-sample GC max outlier 当 RC）+ B7（NetworkDelay 决定性 SQL 维度漏跑）+ A10（state 双自创）。详见 Section 8
- **共性 vs sonnet-specific**：本 case 不在 53 case 列表 → sonnet-specific failure。**跨 case sonnet 模式三次重现**：case 1495 R18 主动 dismiss travel-plan GT 信号 + case 4229 R18 没合成 basic-service survivorship signature + case 2682 R12 看到 travel-plan 33.8x 没锚反找下游 = sonnet 在"看到 GT 顶点信号但因下游解释 / cascade rationalize 不锚定"上的系统性盲点
- **给 sonnet 中间件设计的提示**：A2+A11 强化版（GT 顶点 + chaos 物理机制 prior）+ B7（NetworkDelay/Loss 双面 span 时差 SQL）（详见 Section 9）；这是 NetworkChaos 类 case 的关键缺口
