### 3. chaos 机制

**chaos 物理机制**：tc qdisc 在 ts-seat-service pod 网卡上对 → ts-config-service 方向限速 828 KB/s + buffer 仅 1447 字节。HTTP request 的 header + body 体积超过 1447 字节 → packet drop → seat 发起的 config 调用 socket send 阻塞 / fail。

**预期因果链**：
1. seat→config TCP 连接的 send buffer 立即满 → seat 端 syscall write() 阻塞或 EAGAIN
2. seat 的 service code 调 config 时挂起（HTTP client 等不到 response）
3. seat 的 controller method（如 `SeatController.getLeftTicketOfInterval`）卡在等 config 响应，不返回
4. **seat 的 controller 持有的资源全被锁死**：DB connection、HTTP connection、thread pool 都不释放
5. 上游业务流（travel / travel2 / travel-plan / preserve）调 seat → seat controller 不响应 → 上游 controller 也卡住
6. travel2 controller 持有的 HikariPool DB connection 因为 controller 不返回也不释放 → HikariPool **看上去** 耗尽 → 后续请求等 connection 30s timeout 抛 `HikariPool-1 - Connection is not available`
7. cascade 末端 (loadgenerator → ui-dashboard → travel-plan → travel2) 都看到 20-30s 超时 + 高错误率

**关键因果区分**：HikariPool 报的 `Connection is not available, request timed out after 30002ms` 是 **HikariPool wait timeout**（等 connection 30 秒没拿到），**不是 DB 真的故障 / 拒绝服务**。connection 被前一批 controller 持有不释放（因为 controller 在等 seat），新请求拿不到。这是 cascade 的 victim 现象，不是 RC。

### 4. 调用树（normal 期实测）

```
loadgenerator
  └─ ts-ui-dashboard
      ├─ ts-travel-plan-service
      │   └─ ts-route-plan-service
      │       └─ ts-travel2-service        ◀── sonnet & qwen 都锚定这里 (cascade 末端 victim)
      │           └─ ts-seat-service       ◀── GT (chaos source)
      │               └─ ts-config-service ◀── GT (chaos target)
      ├─ ts-travel-service
      │   └─ ts-seat-service ──────────────┘
      └─ ts-preserve-service
```

**关键拓扑事实**：seat-service 是 trainticket 业务流的核心汇聚点。preserve / travel / travel2 / travel-plan 四条独立业务流的 child span 调用关系交集都包含 seat-service。seat→config 是系统第二高频 cross-service edge（normal 2299 calls/4min，仅次于 loadgen→ui-dashboard）。seat 卡住 = 全系统瘫痪。
