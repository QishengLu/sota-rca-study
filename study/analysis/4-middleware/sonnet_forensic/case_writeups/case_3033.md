### 3. chaos 机制

HTTPResponseReplaceCode 在 **caller 侧**替换 HTTP 响应码：当 ts-food-service 向 ts-train-food-service 发出 `GET /api/v1/trainfoodservice/trainfoods/*` 请求时，chaos framework 在响应从 ts-train-food-service 返回的路径上将 status code 替换为 503（status_code=6）。

关键因果链：ts-train-food-service 服务端正常处理请求 → 正常返回 200 → chaos framework 在 caller 侧将响应码替换为 503 → ts-food-service 接收到 503 → ts-food-service 的 `GET /api/v1/foodservice/foods/...` endpoint 触发 `BasicErrorController.error`（Spring Boot 默认错误处理）→ 向上游返回 500。

GT 包含两个服务：ts-food-service（Error 请求的直接受害者）+ ts-train-food-service（chaos injection_point 的 server 端，response code 被替换的目标边）。注意 ts-train-food-service **自身从不知道** chaos 存在，其 server-side span 永远显示正常的 200。

### 10. 判断

- **数据是否完整**：✅ ts-food-service Error 信号明确，edge ratio ≈ 1.0 是 ReplaceCode 直接指纹
- **chaos 是否生效**：✅ ts-food-service 99.8% error rate，BasicErrorController 417 次触发
- **真假盲区分类**：**(b) 框架/模型盲区** — 数据有完整信号，caller-callee status 不对称（caller 417 Error，callee 418 × 200）是 HTTPResponseReplaceCode 的教科书指纹；sonnet 识别了不对称但得出了错误推论（排除 callee 而非锁定 callee）
- **失败模式归纳**：A13（callee-healthy 当 callee 无辜的反证，排除了真因）→ A12（节点物理拓扑因果假说，因果反向）→ A3（cores 误读为百分比，无效锚定 ts-seat-service）→ A10（自创状态标签）
- **共性 vs sonnet-specific**：全部 sonnet-specific；qwen3.5 直接锁定 ts-food-service 答对，未经历 callee 排除路径
- **给 sonnet 中间件设计的提示**：A13 + A12 组合是本 case 最高价值干预点；A13 需要一条"caller-side chaos 类型下 callee 200 的反解读规则"（"如果 caller 接收到 Error 但 callee server span 全为 200，这是 caller-side response manipulation 的签名，不能排除 callee 为 GT"）；A12 需要"节点假说必须先验 cpu_limit_utilization < 50% → 节点 CPU spike 是症状非 root"的反向校验。
