### 3. chaos 机制

HTTPRequestDelay 在 ts-preserve-service 这一侧的 HTTP client（OkHttp / RestTemplate）上拦截**发往 ts-seat-service:8080 POST /api/v1/seatservice/seats 的 outbound HTTP request**，在请求发出前阻塞 3308ms。物理表现：

- preserve 的 client thread 在 `restTemplate.exchange(seat_url, ...)` 这行卡住 3.3 秒（等待 chaos sidecar 放行）
- request 发出去之后，server 端（ts-seat-service）正常处理，毫秒级返回 response
- 因此 trace 上看到：**preserve 的 PreserveController.preserve span 长 ~3.7s，但其 child span (POST seat-service) 只有 9ms**——差值 ~3.3s 就是 chaos delay（这部分时间被消耗在 preserve 的 client 线程等待 chaos sidecar 放行，**不会产生任何 OpenTelemetry trace 记录**，因为 trace 只追踪 HTTP 发出去之后的 RTT）
- server 端 (seat-service) 自己的 endpoint duration **完全不变甚至更快**（因为 abnormal 期 throughput 降低，server 负载下降）
- causal_graph 把 RC 标在 `ts-preserve-service`，因为 chaos 在 preserve 这一侧；GT 字符串列入 seat 是因为它是 chaos 边的另一端

### 10. 判断

- **数据是否完整**：✅ 完整（preserve service avg 4.2x + endpoint 3.9x + parent-child 时差 237x + chronic AMQP check 数据齐全）
- **chaos 是否生效**：✅ 强生效（preserve 端点 3677ms vs normal 944ms 增量 2733ms ≈ delay_duration 3308ms）
- **真假盲区分类**：(b) **框架/模型盲区**——数据有强信号且 sonnet 在 R31 已经看到（preserve→seat 14ms→3324ms 237x），但 chaos 物理机制层先验缺失导致解读为 client 内部 untraced blocking → hypothesis-fitting 走偏到 chaos 链外
- **失败模式归纳**：A（HTTPRequestDelay 物理机制因果链解读错误）+ B（强证据 anchor 持久性缺失，R31 被新假说覆盖）+ C（对照组测试系统缺失：chronic + metric 反证 + 时间显著性三重对照组都没做）+ D（自创 schema 外 state 标签 UNAVAILABLE，跨 case 第三次）
- **共性 vs sonnet-specific**：**纯 sonnet-specific failure**——4 条根因 qwen3.5 都没犯。两 model 唯一共性是都没识别 HTTPRequestDelay 物理机制；qwen3.5 走对纯属浅启发偶然命中（"Error span 唯一来源"路径），不是系统能力
- **给 sonnet 中间件设计的提示**：A + B + C 三条组合（详见 Section 9）。A 在 chaos 物理机制层建立先验阻断假说源头，B 锁定 R31 强证据防止新假说覆盖，C 在锚定前强制对照组 check 暴露 chronic noise 和 metric 反证。三条都锚定 chaos 物理机制 → 预期因果链信号的对应关系，不依赖统计 ratio 阈值
