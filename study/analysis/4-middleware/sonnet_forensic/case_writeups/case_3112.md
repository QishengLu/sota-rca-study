### 3. chaos 机制

ChaosBlade NetworkPartition 在 pod 网络命名空间下 iptables/tc DROP 来自/到 target service 的所有数据包（direction=both 双向）。物理效果：
- ts-preserve-service 调 ts-seat-service 的 TCP 连接全部被 DROP（SYN 出去无 SYN-ACK 回来 / 连接维持但数据包丢失）
- preserve 端 outbound HTTP CLIENT call 卡在 connect timeout / read timeout（默认 130s 量级，到达 RestTemplate / OkHttp 默认 timeout）
- seat 端 server 不收到任何来自 preserve 的请求；其他 caller (travel/travel2/travel-plan/seat-self) 仍能正常调 seat

预期因果链：
1. preserve→seat outbound CLIENT span：130s wallclock 后 TCP timeout / 5xx return；该 CLIENT span 无 child server-side span（seat 服务从未收到请求）
2. preserve 自身 endpoint POST `/api/v1/preserveservice/preserve` 在 step 调 seat 的位置 hang 130s → ts-ui-dashboard timeout 20s 返 503/500
3. seat 自身 server-side spans 来自其他 caller 仍正常处理，ts-seat-service service-level avg 几乎不变（但 chaos 仍 affects seat：preserve→seat edge=0 是它的 chaos signature）

NetworkPartition 标准指纹（按 skill 库）：edge ratio 0.3-0.5 + jvm.cpu 15x + hubble NaN —— 本 case 是更极端版本：**edge ratio = 0.000**（128→0 完全阻断），preserve service-level avg ratio 110x。

#### sonnet 自己看到的 NetworkPartition 物理 signature（被错读）

R29 think 原话：

> "the 130-second 'POST' span from ts-preserve-service has no child spans in any service... attr_http_response_status_code: null (NO HTTP response received!)... attr_http_request_content_length: null"

**这就是 NetworkPartition / TCP blackhole 完美物理签名**：outbound CLIENT span wallclock 130s + 无 server-side child span + http_response_code 空 + content_length 空 = TCP 数据包被中间层 DROP，连接维持但永不返回。sonnet 写出了所有 fingerprint 但没识别为物理层 chaos——他在 R34 转向 "food-service 因 AMQP error hang" 的应用层假说。

### 9. 中间件提示维度（针对本 case 4 条失败模式 1:1 设计）

| 失败模式 | 中间件提示维度 | 触发时机 |
|---|---|---|
| A4 + A11 + A5 | **caller 端报错主谓反向 + chronic baseline 强制对比**（跟 case 1280 同条目）：当 log 文本含 "X: AmqpIOException / UnknownHostException / Connection refused / Name or service not known" 时强制 reporter=caller / X=被报告对象，锚定 X 之前必须验证 X 自身 ≥ 1 维度有 ≥ 2x 直接证据；同时强制跑 normal vs abnormal log pattern count 对比，ratio < 2x 或 normal ≥ abnormal → chronic dismiss。本 case `Failed to check/redeclare` 1.06x、`ts-rabbitmq` 0.78x 任一对比就能否决 rabbit 假说 | caller-side error log 字面命名某 service 时 + 任何 log pattern 进入"证据"角色之前 |
| B1 + B6 | **NetworkPartition / outbound 物理签名 SQL 模板**：识别到 "outbound HTTP 长 timeout（>= 30s）+ 无 child span + http_response_code 空" 任一现象时强制：(1) 跑 caller-callee edge ratio SQL（JOIN normal/abnormal_traces ON parent_span_id=span_id GROUP BY caller/callee），ratio < 0.1 是极端 drop / 0.3-0.6 是中度 drop，两端 service 直接进 GT 候选；(2) 跑 caller-side outbound CLIENT span subquery（`span_kind='Client'` + 无对应 child server-side span + 按 caller-callee 对 GROUP BY），无 child 的 caller-callee 对就是被阻断边。本 case 跑 SQL 1 行 preserve→seat 128→0 直接命中 GT | trace 中出现长 timeout outbound span 或 "no child span" 现象时 |
| A11 chaos 物理机制 prior | **outbound HTTP 长 timeout 物理签名识别**：trace 中出现 "POST/GET span wallclock ≥ 30s + http_response_code null + content_length null + 无 child span" 四件套时，必须识别为网络层 DROP / blackhole（NetworkPartition / NetworkBandwidth 极端 / DNSRandom），不接受 application-level hang 解读（业务 hang 不会让 outbound HTTP 层 timeout，HTTP 客户端有独立 timeout）。timeout 量级线索：130s ≈ RestTemplate / OkHttp 默认 connect + read timeout，60s ≈ Spring WebClient 默认；对应 chaos 候选不同。然后强制跑 B1+B6 SQL 定位被阻断边的两端 service | 任何 outbound span ≥ 30s wallclock 时 |
| A10 | **state vocabulary 强制小写词表**（跟 case 1280/1495 同条目）：state 输出必须从 causal_graph schema 词表选取（小写 `high_cpu` / `high_memory` / `high_gc_pressure` / `high_avg_latency` / `high_error_rate` / `injection_affected` / `restarting` / `timeout` / `healthy` / `unknown`），拒绝大写自创 `UNAVAILABLE` / `HIGH_CPU` / `HIGH_LATENCY` 等 | final answer 输出前 |

**最有效组合**：A11 + B1+B6——单条物理签名识别（outbound 130s + 无 child span + http_response null = 网络层 DROP）+ 一条 edge ratio SQL（preserve→seat 128→0 直接命中两端 GT）就能扭转本 case；跟 1280 共享的 A4+A5 是次要安全网。

### 10. 判断

- **数据是否完整**：✅ 数据齐全，NetworkPartition 完美 signature（edge ratio 128→0、preserve avg 110.7x、outbound 130s 无 child span 物理签名）
- **chaos 是否生效**：✅ 完全生效（双向阻断完美命中：preserve→seat edge=0，seat→preserve 反向也无效）
- **真假盲区分类**：(b) 框架/模型盲区——sonnet **写出了**完整物理 signature 但用应用层假说反向解读；同时重蹈 case 1280 caller log 字面解读模式
- **失败模式归纳**：A4+A11+A5（caller log 字面 + chronic check 缺失，重蹈 1280 模式）+ B1+B6（NetworkPartition 核心 SQL 维度漏跑）+ A11 chaos 物理机制 prior 缺失（写出物理 signature 但反向解读）+ A10（state 自创）。详见 Section 8
- **共性 vs sonnet-specific**：本 case 不在 53 case 列表 → sonnet-specific failure。**跨 case sonnet 模式**：连续两个独立 case (1280 + 3112) 都因 caller 报告的 "X 不可达" 字面解读 → 锚 rabbit；且 3112 中 sonnet **物理签名识别能力存在但因果机制 prior 缺失**——他能描述 outbound TCP timeout 现象但解读为应用层 hang
- **给 sonnet 中间件设计的提示**：A11 chaos 物理机制 prior + B1+B6 NetworkPartition SQL 模板（详见 Section 9），是 sonnet 在 NetworkChaos 类 case 上的关键缺口
