### 2. GT 注入

```
fault_type: 5 = HTTPRequestAbort
display_config:
  injection_point:
    app_name: ts-travel2-service       # 注入到 caller (做 outbound HTTP call 的一方)
    method: POST
    route: /api/v1/seatservice/seats/left_tickets
    server_address: ts-seat-service    # callee（请求被 abort 永远到不了这里）
    server_port: 8080
  duration: 4
ground_truth.service: [ts-travel2-service, ts-seat-service]
```

### 3. chaos 机制

HTTP-request-abort 是 **caller-side chaos**：拦截 ts-travel2-service 发往 ts-seat-service `POST /seatservice/seats/left_tickets` 的 outbound HTTP request，在 connection 建立或刚发出阶段直接 abort（I/O error: Connection reset / refused）。

物理后果：
- **caller (ts-travel2-service)** 看到大量 outbound CLIENT span 出现 `status_code='Error'` + `http_response_status_code=null`（没收到 response）+ `no child span`（callee 没产生子 span）
- **callee (ts-seat-service)** **完全 silent on errors**：因为 request 根本没到，seat 没产生 error span / error log，trace count 反而 -51%（流量被 caller 端吞）

GT 共病的合理性：travel2（caller，看到 outbound failure）+ seat（callee，被流量切断），两端都被影响但**信号方向相反**（caller side 满是 error，callee side 完全 silent）。这是个高陷阱 case：sonnet 容易把 caller 看到的 SEVERE log "ResourceAccessException I/O error" 当作 callee 的故障证据，反向追到 callee 的 callee。

### 6. sonnet 完整推理路径

sonnet 跑 38 round → 锚 `ts-basic-service` HIGH_CPU ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R3-5 | discovery + service-level error rate | 识别 travel-plan / route-plan / loadgen / ui-dashboard 是 cascade（30%/25% error rate） |
| R7-10 | trace drill | 顺 trace 走调用链，识别 travel-plan → route-plan → travel2 |
| R14 | trace 单条精读 | 看 trace `7c60a59...` 完整链路，看到 travel2 Error span |
| 🟡 R17 | **看到 GT 指纹但没识别** | "ts-travel2-service POST span: status_code=Error, http_response_status_code=null, no child spans found from this POST" — **这正是 HTTP-request-abort 的教科书指纹**（caller 端 outbound abort），sonnet 看到原始证据但没把它当 RC 信号 |
| R19 | 调用关系扩展 | "ts-travel2-service calls ts-basic-service and ts-seat-service" — **从这里开始 sonnet 把注意力下移到 callee** |
| 🔴 R22 锚定瞬间 | metric drill | "**ts-basic-service queueSize: NORMAL=9, ABNORMAL=38.75, ratio=4.31x — by far the biggest metric anomaly**" — 用 ratio-only 锚定 |
| R25 | 反证检测 | "many other services also have high queueSize in abnormal: ts-route-service 118.75, ts-station-service... ts-basic-service NORMAL queueSize=9（低基数）" — **自己已发现 ratio 是除以小基数造成的视觉 outlier**，但**没回退** |
| R28 | 加固 | 比较 NORMAL queueSize top services（route 173.5, config 111.5），仍坚持 ts-basic-service |
| 🟡 R32 | **第二次看到 GT 指纹** | "failing POST spans from ts-travel2-service are CLIENT spans, attr_http_response_status_code=null, no HTTP response received" — **再次原始指纹**，仍没切换 anchor |
| R35-37 | 加固合理化 | "ts-travel2-service logs: [getTicketsByBatch][before get basic] — service gets trips first then calls basic" → **强行因果建构**：travel2 报错 → 因为 ts-basic-service queueSize 高 → ts-basic 是 RC |
| R38 + FINAL | 锁定 | ts-basic-service HIGH_CPU |

**sonnet 失败点（具体行为）**：
1. **看到正确 GT 指纹两次但没切换 anchor**（R17 + R32 都明确捕获 "CLIENT span + no HTTP response + no child span" — HTTP-request-abort 教科书指纹）
2. **ratio-only fallacy（除以小基数视觉 outlier）**：用 queueSize ratio 4.31x 当主证据，没看绝对值（38.8 排第 5，比 route/station/travel/auth 都低）
3. **反证检测但不回退**：R25 自己发现 baseline=9 是低基数，但仍锚定 ts-basic-service
4. **silence-as-causation 倒因果**：caller-side 报错（ts-travel2 outbound failure）→ 推 callee 是 RC（ts-basic-service）。但 chaos 是 caller-side 注入，跟 callee 健康无关
5. **没做 caller-callee edge ratio drop SQL**：(ts-travel2 → ts-seat) 0.003 是 GT 最干净指纹，全程没跑
6. **没系统看 endpoint-level Error 分布**：ts-travel2 POST /trips/left 80% error rate（229/286）是端点级直接证据，没系统跑
7. **anchor 锁死后过度合理化**：R22 锚定后 13+ round 全用在合理化，反复强行建构 travel2 → basic 的因果链

### 10. 判断

- **数据是否完整**：✅（ts-travel2 outbound CLIENT span Error 855 + edge ratio 0.003 + endpoint 80% error rate 三个维度都有强信号）
- **chaos 是否生效**：✅（caller side 855 status_code Error / 285 SEVERE log；callee side trace count -51%；edge drop ratio 0.003）
- **真假盲区分类**：**(b) 框架/模型盲区** —— 数据有大量直接 GT 信号（caller outbound abort 教科书指纹 + edge ratio + endpoint Error 分布），sonnet 实际**两次看到指纹原文**（R17+R32）但没识别；anchor 被 ratio-only outlier 吸引锁死
- **失败模式归纳**：caller-side outbound abort 指纹识别失败 + ratio-only fallacy + 反证检测不回退 + silence-as-causation 倒因果 + anchor 锁死后过度合理化
- **共性 vs sonnet-specific**：**sonnet-specific failure** —— qwen3.5 命中 ts-seat-service ✓；sonnet 是被 ts-basic-service queueSize 4.31x 视觉 outlier 锁死。**跨 case 已观察到 sonnet 稳定的"反证检测但不回退"决策风格**（case 1140 + case 2830 都犯）。
- **给 advisor 的提示**：
  1. caller-side outbound abort 指纹强规则（CLIENT span Error + no HTTP response + no child span → caller GT）
  2. ratio-only 反例校准（除以小基数的视觉 outlier 必须 dismiss）
  3. 反证强制回退 + anchor 锁死预算上限（已发现矛盾证据时强制 reset）
  4. caller-callee edge ratio drop 维度作为 HTTPChaos 类无条件触发动作
