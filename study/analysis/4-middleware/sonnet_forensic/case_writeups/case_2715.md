### 3. chaos 物理机制 + 预期因果链

NetworkBandwidth 用 `tc qdisc tbf` 在 source pod 网卡上限速。`direction=to` 限制 ts-station-service pod 收来自 ts-basic-service 的字节流。**limit=138 字节** 是关键——HTTP response 头部最小也要 100+ 字节，limit 138 字节意味着哪怕 HTTP 200 OK 的小 response 都过不去。

**预期因果链**（按物理机制推）：

```
[chaos] tc tbf 限制 station pod 入站 (来自 basic) 字节流，limit=138B 极端
   │
   ▼
[直接症状] ts-basic→ts-station 这条边的 caller-side client span 收不到 response，
           SocketTimeout；OTel SDK 长 timeout/失败 client span 不正常 export
   │
   ▼
[二阶 cascade] basic 调 station 的 thread 被卡 30s+，basic 自己的 thread pool 被这些
              卡住的 thread 占满 → basic 自己 incoming 请求也几乎全部卡住（thread
              starvation）→ basic 上游 caller (travel) 等不到 response 也被卡
   │
   ▼
[三阶 cascade] travel 调 basic 的 thread 阻塞 30s → travel 自己的 HikariPool DB
              connection 被这些卡住的 thread 占满（每个 request 占用一个 DB conn）
              → HikariPool exhaustion → "Connection is not available" SEVERE log
   │
   ▼
[SLO 末端] loadgenerator/UI 看到 20s timeout (ts-ui-dashboard 客户端 timeout)
```

GT = ts-station + ts-basic 双服务（chaos source/target 两端都在因果链最上游）。

### 4. 调用树（normal 期实测）

```
loadgenerator
  └─ ts-ui-dashboard
      └─ ts-travel-service::POST /trips/left
          └─ ts-basic-service::POST /basic/travels             ◀── chaos basic→station 这条边被卡
              └─ ts-station-service::POST /stations/idlist     ◀── chaos target (direction=to)
                  └─ (DB / cache)
              └─ ts-train-service / ts-route-service / ts-price-service
```

ts-travel 是 ts-basic 的 caller（不是 chaos source），是 cascade 的**第三层 victim**。

#### 5.3 ts-travel HikariPool log（sonnet 锚定的伪 RC）

ts-travel-service abnormal SEVERE/ERROR 总共 10 行，dedup：

| message | count |
|---|---|
| `HikariPool-1 - Connection is not available, request timed out after 30000ms.` | 4 |
| `Servlet... CannotCreateTransactionException... Unable to acquire JDBC Connection` | 3 |
| `Servlet... DataAccessResourceFailureException... Unable to acquire JDBC Connection` | 2 |
| `HikariPool-1 - Connection is not available, request timed out after 30001ms.` | 1 |

chronic check：normal 期 0 行 → incident-only ✓

**但是**：incident-only ≠ RC。HikariPool 是 cascade 第三层的二阶症状：
1. ts-travel 调 ts-basic 的 thread 阻塞 30s（因为 basic 调 station 被卡）
2. 每个 ts-travel request 占用 1 个 HikariPool DB connection（Spring Data JPA 默认行为）
3. ts-travel HikariPool size 默认 10，10 个并发 request 等满 30s → HikariPool exhaustion
4. 第 11 个 request 的 30s 等连接超时报 "Connection is not available"

**因果方向**：chaos → basic→station edge 卡 → travel→basic edge 卡 → travel HikariPool 耗尽。HikariPool log 在因果链**末端**，不是源头。

### 6. sonnet 完整推理路径

sonnet 跑 ~11 effective rounds → final pred = `ts-travel-service` UNAVAILABLE/HIGH_ERROR_RATE ❌

| Round | 行为 | 关键决策 |
|---|---|---|
| R2 | 看 service log levels | 发现 ts-travel-service 5 SEVERE + 5 ERROR + 5 WARN（"critical!"）|
| R3 | 查 ts-travel SEVERE log 内容 | 看到 "HikariPool-1 - Connection is not available, request timed out after 30001ms" + "Unable to acquire JDBC Connection" |
| 🔴 R3 锚定瞬间 | think_tool reflection | "Root Cause Identified: ts-travel-service has a database connection failure!" |
| R4-R7 | 查 ts-travel2 / ts-preserve trace | 注意到这些 trace 也 20s timeout，都"卡在某处"——但没把这条线索回推到 chaos 源 |
| R8 | 查 cpu.usage 对比 | 看到 basic CPU max 0.886→0.977 (+10%)、travel CPU max 0.748→0.853——dismiss 为"小幅"（实际 basic avg cpu **跌** 0.35x，sonnet 看的是 max 没看 distribution）|
| R9 | 查 db.client.connections.* | 找到 ts-travel `db.client.connections.timeouts=3` at 10:47:34 + `pending_requests=1` |
| R10 | reflection | "definitive evidence: ts-travel-service has 3 connection timeouts" — 把 db pool timeout 当 RC 充分证据 |
| R11 final | think | "Database Connection Pool Exhaustion (Root Cause)" — 写最终答案 |

**sonnet 失败点**：

1. **R3 first-error log 锚定**：HikariPool/JDBC log 是 SEVERE 等级最响的信号，sonnet 没问"为什么 ts-travel 这一刻 DB pool 耗尽"就锚定。
2. **没问因果上游**：HikariPool 耗尽必有上游原因（每个 connection 被占着干嘛？Spring JPA 默认每 request 占一个 conn → 这些 request 在等什么？）。sonnet 没沿因果链反向推。
3. **R8 max-only fallacy + 误读 CPU 方向**：basic CPU max 0.886→0.977 (+10%) sonnet 当"小幅"，没看 avg 0.245→0.086（**跌 65%，反向**）—— avg 跌反而是 thread starvation 的强信号。
4. **caller-callee edge ratio 全程未跑**：57 round 中没出现 `(t2.service caller, t1.service callee, COUNT)` 形式的 SQL——chaos 最干净指纹。
5. **chaos 物理机制候选未自主推导**：trajectory 中没出现 "NetworkBandwidth / 限速 / tc / 网络层异常" 等候选枚举的 think_tool reflection——agent 凭手里征兆完全可推但没推。
6. **ts-station / ts-basic 自身没单独查证**：ts-station service avg 1.1x、ts-basic 5 条 trace + 0 ERROR log，sonnet 没意识到这种"silent + count 大跌"是 thread starvation 类 chaos 的指纹（不是健康）。

### 10. 判断

- **数据是否完整**：✅ trace/log/metric 完整，足以诊断（caller-callee edge 1263→1、CPU 反向下降、调用链 silent 模式都齐全）
- **chaos 是否生效**：✅ 完全生效（(basic→station) edge -99.9%、basic 自身出站全断、travel cascade -100%）
- **真假盲区分类**：**(b) 框架/模型盲区** — 数据有充足信号，两 model 都没引导自己回到 chaos 物理机制层
- **失败模式归纳**：A（first-error log 锚定 + 没问因果上游）+ B（cascade 末端 victim 锚定 / silent vs loud 区分缺失）+ C（caller-callee edge ratio 矩阵未审计）+ D（chaos 物理机制候选未自主推导）
- **共性 vs sonnet-specific**：**A/B/C/D 全部是两 model 共偏离环节**——qwen3.5 baseline 同样答 ts-travel-service ❌（行为级失败模式见 OBSERVATIONS 描述部分）。该 case 暴露的是 NetworkBandwidth 类 chaos 在两 model 上共有的 "loud victim trap"——不是任何 model 独有
- **给 sonnet 中间件设计的提示**：D + C + A 三条组合（详见 Section 9），用 "从征兆自主推 chaos 候选" 驱动因果链反向追溯，避免"最响信号 = RC"启发式锚定 cascade 末端 victim
