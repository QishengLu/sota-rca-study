### 3. chaos 机制

ContainerKill 把 ts-travel-plan-service 容器 kill -9，K8s 自动重启，container.restarts 0→1（abnormal 期 pod `ts-travel-plan-service-5b7fb74b4f-w8nlh`，**全局唯一新增**——chronic ts-ticket-office-service 3 不变）。重启期间约 10 秒：container.cpu.usage 在 kill 瞬间 = 0（旧容器死亡），重启后 cold-start 飙升至 avg 0.673（normal 0.147，**4.59×**）；上游 ts-ui-dashboard reverse proxy 收到 `dial tcp 10.103.180.158:8080: connect: connection refused` → 返回 502 给 loadgenerator。

注：本 case SQL 1 service-level avg ratio **全部下降**（abnormal trace count 反而 1.5–1.6×，service avg 0.6–0.9×）——abnormal 期负载更高 + 短窗口失败请求未 export 导致幸存者偏差，无法用 service-level ratio_avg 排名定位 GT。必须用 restart counter / container.cpu / hubble NaN 直接指纹。

A2 处理方式：发现 restart counter 0→1 + container.cpu=0 + cpu.time reset 三件套时强制 commit 为 RC，禁止后续假说覆盖。

A5 处理方式：锚定 GC pressure 服务前必须做 normal vs abnormal GC max 对比 SQL；normal 期已有 1s+ GC max → chronic，dismiss。

A3 处理方式：单点采样 CPU 异常不能作为 RC 证据；要求 distribution（avg / p50 / p99）支持。

**新模式 chaos-kill-as-victim 处理方式**：发现 ContainerKill / PodKill 三件套（restart += 1、cpu=0、cpu.time reset、startup log、hubble NaN）后，强制结论"该 service 即 RC，无需查 upstream cause"；禁止把 kill 当成 victim 反向找 upstream。本失败模式与 case 1948 sonnet 的"cascade 方向倒置（callees 当 RC）"是不同方向的同一根因——都源于"找 upstream cause for chaos signature"这种因果链先验缺失。

- **数据是否完整**：✅ restart 0→1、container.cpu=0、cpu.time reset、startup log 都可查；sonnet R16–R22 已实际看到

- **失败模式归纳**：A2（强证据未 sticky）+ A5（chronic GC 锚定，未做 normal vs abnormal 对比）+ A3（单点采样 CPU 锚定）+ **新模式 chaos-kill-as-victim**（chaos kill 当 victim 找 upstream cause）
- **共性 vs sonnet-specific**：sonnet-specific（qwen3.5 正确）；本 case 失败模式与 case 1948 sonnet 是**同根因不同方向的因果链倒置**——
  - case 1948 sonnet：找下游 callees 当 RC（cascade 方向倒置）
  - case 4739 sonnet：找 upstream cause 当 RC（chaos kill 当 victim）
  - 两者共因：缺"chaos signature 自身即 RC"先验
- **给 sonnet 中间件设计的提示**：A2 + 新模式 chaos-kill-as-victim 组合——(1) restart 0→1 + cpu=0 + cpu.time reset 任一命中即 lock-in 该 service 为 RC，禁止反向找 upstream；(2) 加 chaos-direct-injection 先验卡：ContainerKill / PodKill / PodFailure 类 chaos 不存在 upstream cause，发现 kill signature 立即终止上游搜索
