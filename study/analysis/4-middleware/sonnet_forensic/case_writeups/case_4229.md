### 3. chaos 机制

ChaosBlade NetworkPartition iptables/tc DROP basic↔travel 之间的数据包。注意调用拓扑实际方向：**ts-travel-service → ts-basic-service**（travel 是 caller，basic 是 callee；chaos `direction='to'` 是 chaos 配置内部参数，不等于调用方向）。

预期因果链（按 OBSERVATIONS 记录的 NetworkPartition 指纹）：
1. travel 调 basic 的 outbound HTTP CLIENT span 被 packet drop → connect/read timeout → travel 端 thread 阻塞 + retry 风暴
2. travel 端 jvm.system.cpu.load_1m 暴涨（D-state thread + busy retry thread 综合）—— **15.97x 是 chaos 真信号顶点**
3. basic 端被动接收 throughput 下降，self-cpu 弱升（4.11x）
4. 上游 caller chain（route-plan / travel-plan / ui-dashboard）throughput 全降，cpu 因 retry 雪崩跟着升 5-15x（cascade，不是 chaos 直接信号）
5. **关键 survivorship 指纹**：GT 服务 service avg **反而更快**（basic 0.76x / travel 0.78x）—— 失败 trace 没 export，残存的是没经过 chaos 路径的快请求；trace count 跟全集群同步 -60% 左右（不极端）

NetworkPartition 标志组合（OBSERVATIONS case 1140/283/4229 共同结论）：**avg 反而 < 1.0x + trace count 跟全系统一致 (-50~-65%) + jvm.system.cpu.load_1m 暴涨 + hubble 全 NaN**。

### 8. 失败模式

| 标签 | 本 case 具体表现 | qwen3.5 对应 |
|---|---|---|
| **A2 + 新模式（NetworkPartition signature 合成缺失）**（强证据 dismiss + 多信号未合成） | R18 看到 ts-basic-service "Actually BETTER in abnormal" (avg 0.76x) + R19 看到 ts-basic-service container.cpu **3.7x avg / 5.7x max**——这两条独立信号合在一起就是 NetworkPartition survivorship 完美 signature（avg 反而下降 + cpu 暴涨 = 失败 trace 没 export 残存快请求 + retry 风暴烧 cpu）。sonnet 各看一次但没回头合成；写 R19 时甚至没引用 R18 "BETTER" 这个反常现象。chaos 因果机制视角：service avg 0.76x 单看像 healthy，但跟 cpu 3.7x 共存只可能是"失败请求被吞 + 残存请求是没经过 chaos 路径的快请求 + 被 retry 烧 cpu"——这是 NetworkPartition 独家组合。**新模式候选**：信号合成层失败，需要把 "service avg 反而下降 + cpu 升 + count 跟全集群一致" 三件作为 NetworkPartition 强制识别规则 | **共性弱**：qwen baseline 没走到 specific metric 这一步，没机会合成；sonnet 比 qwen 走深一层但合成失败 |
| **A3（system-wide CPU 假说 + 没区分 chaos source 与 cascade）** | R20-21 看到所有 6 个 service 的 cpu 都 5-16x → 直接归因 "system-wide HIGH CPU"。chaos 物理机制视角：NetworkPartition 不会引起 system-wide CPU 直接饱和——它阻断 (basic↔travel) 一条边；其他 service cpu 升是 cascade retry 雪崩传导（caller 在 travel 上 timeout retry 占 thread + 其他 service 跟着 thread 阻塞）。区分 chaos 直接信号 vs cascade 的方法：**jvm.system.cpu.load_1m 排序**——travel 上 15.97x（chaos retry 源最强）+ basic 上自己 jvm.cpu 较弱（callee 被动），其他 cascade 服务 jvm.system.cpu.load_1m 都不及 travel。sonnet 没跑 jvm.system.cpu.load_1m 的全表排序——只查了 container.cpu.usage 和 jvm.cpu.recent_utilization | qwen 路径相似（也没区分 chaos source vs cascade，沿 cpu / latency 排名锚某个 cascade 服务）|
| **A11（trace duration gap 物理机制误读）** | R44 ts-seat-service POST 597ms - 唯一 child (ts-order-other-service) 21.9ms = 575ms unaccounted gap。chaos 物理机制视角：seat 在该 trace 中调 travel-service（travel 是 chaos GT），travel 端 retry 风暴让 seat 的 outbound CLIENT span 等 575ms。sonnet 把 unaccounted gap 当 "seat 自己 hang" 的证据 → 锚 seat。反查缺失：本应查 seat 该 span 内是否还有调 travel 的 CLIENT span 被 trace exporter drop（seat 调 travel 的 retry/timeout 不一定都成功 export 子 span）+ 看 seat outbound edge 全集群层面是否 partition | qwen3.5 baseline 走"503 + caller 端报错锚 caller 链上服务"路径，没走 trace duration gap 这一层 |
| **B1（caller-callee edge ratio 没跑指向 chaos edge）+ B3 子表现（jvm.system.cpu.load_1m 全表排序没跑）**| sonnet 跑过 service-level metric ratio 全表（R20）但没跑 caller-callee edge ratio SQL（实测 (travel→basic) edge ratio 0.398，单看不极端但跟 jvm.system.cpu.load_1m 15.97x 组合可定位）；也没跑 jvm.system.cpu.load_1m 全表排序（实测 travel 163.70 / 全集群其他 < 30，是绝对顶点；跟 container.cpu 全表 5-16x 中 travel 不显特别但 jvm.system.cpu.load_1m 维度直接命中 GT）。两个 SQL 维度都是 NetworkPartition 决定性查询路径 | qwen 也没跑（共性）|
| **A10**（state vocabulary 自创大写） | 输出 state=`HIGH_CPU`（大写自创），causal_graph 词表是小写 `high_cpu` / `injection_affected` 等 | qwen baseline 也自创 `HIGH_LATENCY`（同问题）|

### 9. 中间件提示维度（针对本 case 5 条失败模式 1:1 设计）

| 失败模式 | 中间件提示维度 | 触发时机 |
|---|---|---|
| A2 + NetworkPartition signature 合成 | **多信号合成强制 + survivorship signature 识别**：reflection 中出现"X service avg 反而下降 / actually BETTER / better than normal" 并且后续看到 X 的 cpu 升高（≥ 2x） + trace count drop 跟全集群一致 (-50~-70%) → 强制识别为 NetworkPartition / NetworkBandwidth survivorship signature（失败 trace 没 export 残存快请求 + caller 端 retry 烧 cpu），X 直接进 GT 候选并必须查 jvm.system.cpu.load_1m + edge ratio。本 case ts-basic-service 0.76x avg + 3.7x cpu + count 0.41x 三件齐全任一组合识别就直达 GT | reflection 看到"service avg 反而下降"现象时 |
| A3（system-wide CPU 假说否决） | **system-wide CPU 假说强制反证**：当多服务 cpu 同时上升（≥ 5 个 service ≥ 3x）时，禁止直接归因 "system-wide CPU pressure"——必须问"这是 chaos 直接信号还是 cascade retry 雪崩"。强制跑 jvm.system.cpu.load_1m 全表降序排序（D-state thread + retry 阻塞最敏感）；ratio top1 是 chaos retry 源最直接候选；跟 container.cpu 综合排序对比，jvm.system.cpu.load_1m 顶点 ≠ container.cpu 顶点时优先信任 jvm.system.cpu.load_1m（cascade 服务 container.cpu 升是 retry 雪崩传导，不是 chaos 直接信号）。本 case travel 上 jvm.system.cpu.load_1m 15.97x = chaos 直接信号 | 多服务 cpu 同时上升、形成 system-wide 假说倾向时 |
| A11（trace duration gap 物理机制识别） | **trace duration gap 因果归属反查**：trace 中出现 "X span duration 远 > Σ child span duration" gap 时（如 597ms - 22ms = 575ms unaccounted），不能直接归因 "X 自己 hang"。反查方向：(1) X 在该 trace 中是否还有 outbound CLIENT span 被 trace exporter drop（HTTP timeout / retry 子 span 不一定 export）；(2) X 调用的下游 service 是否在 chaos signature 上有强信号（jvm.system.cpu.load_1m 顶点 / edge ratio 极低 / restart counter）；(3) chaos 物理机制视角下 X 端 unaccounted time 跟 chaos 类型对应（NetworkPartition timeout 130s 量级 / NetworkDelay 有 delay_duration 量级 / JVMStress 跟 GC pause 量级）。本 case 575ms gap 跟 travel jvm.system.cpu.load_1m 15.97x 直接关联（seat 调 travel 在 retry 风暴中等待） | trace duration gap > 50% 时 |
| B1 + B3 子表现 | **NetworkChaos 类强制 SQL 模板（caller-callee edge ratio + jvm.system.cpu.load_1m 全表排序）**：识别到 hubble metric NaN / outbound HTTP 长 timeout / service avg 反向下降 等 NetworkChaos signature 后强制：(1) 跑 caller-callee edge ratio SQL（JOIN normal/abnormal_traces ON parent_span_id=span_id GROUP BY caller/callee ORDER BY ratio ASC LIMIT 10）；(2) 跑 jvm.system.cpu.load_1m 全表降序（强制 metric IN ('jvm.system.cpu.load_1m')，不能用 LIKE '%cpu%' 漏掉）；两个 SQL 是 NetworkChaos 类的决定性查询路径 | hubble metric NaN 出现 + 多服务 cpu 上升 + service avg 反向下降 任一现象出现时 |
| A10 | **state vocabulary 强制小写词表**（跟前几个 case 同条目）：state 输出必须从 causal_graph schema 词表选取（小写 `high_cpu` / `injection_affected` / `high_avg_latency` / `timeout` / `unknown` 等），拒绝大写自创 | final answer 输出前 |

**最有效组合**：A2 + A3 — A2 强制把"service avg 反而下降"识别为 survivorship signature 直接定位 ts-basic-service（GT），A3 强制 jvm.system.cpu.load_1m 全表排序定位 ts-travel-service 163.70 顶点（GT 另一端）。两条任一触发都能扭转本 case；跟 case 1495 共享 A2 强证据合成机制。

### 10. 判断

- **数据是否完整**：✅ 数据齐全，NetworkPartition 经典指纹（survivorship avg < 1.0x + jvm.system.cpu.load_1m 15.97x + edge ratio 0.398 + cpu 双 GT 4.11x/9.45x）
- **chaos 是否生效**：✅ 完全生效（avg < 1.0x survivorship signature + jvm.cpu 暴涨）
- **真假盲区分类**：(b) 框架/模型盲区 + 共性认知盲区——三 model 都没识别 NetworkPartition signature 组合
- **失败模式归纳**：A2+新模式（NetworkPartition signature 合成缺失）+ A3（system-wide CPU 假说掩盖 chaos 拓扑差异）+ A11（trace duration gap 物理机制误读）+ B1+B3 子表现（NetworkChaos 类 SQL 维度漏跑）+ A10（state 自创）。详见 Section 8
- **共性 vs sonnet-specific**：
  - **共性**：A3 + B1+B3 — 三 model 都没识别 NetworkPartition signature 组合，都没跑 caller-callee edge ratio + jvm.system.cpu.load_1m 全表排序
  - **sonnet-specific**：A2 NetworkPartition signature 合成缺失——sonnet 走得更深拿到了 R18 + R19 双信号但合成失败；qwen 路径短没机会合成；sonnet 是"信号到位但合成失败"，qwen 是"没查到信号"
  - **跨 case sonnet 模式确认**：case 1495 + 4229 共享同一根本病灶 A2——sonnet 实际看到 GT 信号但因主动 dismiss / 信号未合成而失败
- **给 sonnet 中间件设计的提示**：A2 + A3 组合（详见 Section 9）——A2 多信号合成强制（"service avg 反向下降 + cpu 升 + count 跟全系统一致 = NetworkPartition survivorship 三件套"识别规则）；A3 system-wide CPU 假说强制反证（jvm.system.cpu.load_1m 全表排序定 chaos retry 源）
