### 3. chaos 机制

HTTPResponseReplaceCode 在 caller 侧替换响应码。本 case：当 ts-security-service 调用 ts-order-service 的 `GET /api/v1/orderservice/order/security/{checkDate}/{accountId}` 时，response code 被替换为错误码（503）。

因果链：security 在做"安全检查"流程中，**串行**调用两个 order 服务（order-service + order-other-service）以验证账户订单。当第一个调用（→ order-service）返回 503 时，security 抛出 RestClientException → 整个安全检查失败 → 不再调用第二个（→ order-other-service）→ security 向 caller（preserve）返回 500 → preserve 端 100% Error。

GT：ts-security-service（caller，500 错）+ ts-order-service（chaos 注入点的 server，response code 被替换）。ts-order-other-service **不是** RC，它只是 security 短路后未被再调用而已。

### 10. 判断

- **数据是否完整**：✅ caller-callee 不对称信号清晰（security 100% Error vs order-service server 200），edge ratio 1.111 + 0.0 双边对比是 ReplaceCode 串行拓扑教科书指纹
- **chaos 是否生效**：✅ 170 × 100% error，BasicErrorController 全量触发
- **真假盲区分类**：**(b) 框架/模型盲区** — 信号充分；sonnet 看到了 503（caller side）但未识别为 chaos 注入边，反而沿"trace 中断点"反推到一个短路下游
- **失败模式归纳**：A11（missing-span ↔ callee unavailable 等价）→ A9（多维度反证仍锚）+ A2（早期 security 100% error 强证据未 sticky）
- **共性 vs sonnet-specific**：全部 sonnet-specific；qwen3.5 顺着 caller error 直接锁定 security 答对
- **给 sonnet 中间件设计的提示**：A11 + A2 组合是本 case 最高 ROI 干预点；A11 需要"missing-span 因果归属反查"规则（"caller 短路 ≠ callee unavailable，须查 callee 自身 trace count / status / restart 是否真不可用，并问'caller 是否在更早一边已经 fail'"）；A2 需要"早期 ≥ 100% error 服务必 sticky 进 RC，后续假说不能覆盖"。
